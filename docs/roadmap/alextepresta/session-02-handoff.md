## Decisions Made

1. **`vitest.config.ts` excluded from `tsconfig.json`** ‚Äî The `@vitejs/plugin-react` plugin has a type incompatibility with the vitest-bundled vite (rolldown vs rollup). Adding `vitest.config.ts` to `tsconfig.exclude` is the correct fix; vitest uses its own tsconfig resolution.

2. **`CookieOptions` type imported in `server.ts`** ‚Äî TypeScript strict mode required explicit typing on the `setAll` callback parameter. Type is imported directly from `@supabase/ssr` to avoid `any`.

3. **`vi.mock('server-only', () => ({}))` in tests** ‚Äî The `server-only` package throws on import in non-Next.js environments. Mocking it at the top of the test file (before any imports) prevents the test runner from crashing.

4. **No Database generic type parameter on Supabase clients** ‚Äî The typed client requires `supabase gen types typescript` output. This is deferred to Session 3 after the schema is confirmed. All clients use the unparameterized form; Session 3 will regenerate and add the `Database` generic.

5. **Seed uses fixed UUID `00000000-0000-0000-0000-000000000001`** ‚Äî Supabase `seed.sql` has no env var interpolation. This is a dev-only constant; production admin is created through Google OAuth.

## Open Issues

- `next lint` deprecation warning ‚Äî `next lint` is deprecated in Next.js 15.5+ in favor of running ESLint CLI directly. The `check` script works correctly today but will break in Next.js 16. Session 3 or later should migrate: `npx @next/codemod@canary next-lint-to-eslint-cli .`
- The `server.ts` uses `cookies()` from `next/headers` which cannot be tested in unit tests without mocking Next.js internals. Server client tests are omitted by design; integration tests in a real Next.js context cover this in a later session.
- Typed Supabase clients (with `Database` generic) deferred to Session 3.
