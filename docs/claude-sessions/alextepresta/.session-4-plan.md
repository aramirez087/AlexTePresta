## Critical Design Decisions

### Decision 1: Atomic Transaction via PL/pgSQL RPC

**Problem**: Supabase JS client has no native multi-statement transaction API. `createDebt` must insert into `debts` then insert N rows into `installments` atomically. If installment inserts fail after the debt is inserted, the DB would have an orphaned debt.

**Decision**: Write a PL/pgSQL function `create_debt_with_installments` (migration 0004). The TypeScript server action calls it via `adminClient.rpc()`. This follows the exact same pattern as `accept_invite` (0003_invite_accept_fn.sql).

**Alternative rejected**: Application-level compensation (insert debt, on failure delete debt) — not truly atomic; tiny window where a crashed server leaves an orphaned debt.

### Decision 2: Separation of generateForDebt (Pure + DB-Write)

**Problem**: The cron route needs to back-fill installments for existing debts without going through `createDebt`. The PL/pgSQL function is overkill for this case.

**Decision**: `generateForDebt.ts` provides two exports:
1. `computeInstallments(params)` — pure TypeScript, no DB access, fully testable without mocks
2. `generateForDebt(adminClient, debtId, params)` — calls `computeInstallments` then upserts with `ON CONFLICT (debt_id, sequence_number) DO NOTHING`

The `createDebt` server action calls the PL/pgSQL RPC (atomic path). The cron route calls `generateForDebt` (back-fill path). Due date logic lives in TypeScript; the PL/pgSQL function duplicates it in SQL — acceptable because the SQL path is authoritative for creation and the TS path is defensive back-fill.

### Decision 3: bigint in Logic, number at DB Boundary

**Problem**: `database.types.ts` types monetary fields as `number` (Supabase generates these from PostgreSQL types). The operator rules require `bigint` for all monetary values.

**Decision**: 
- Zod schema accepts `number` integers from form/wire (JSON doesn't serialize bigint)
- First line inside server action: convert to `bigint` for all arithmetic
- At DB insert boundary: `Number(bigintValue)` with a comment naming the boundary
- At DB read boundary: `BigInt(row.amount_minor)` before any math

### Decision 4: Migration Renumbering

**Problem**: Session instructions say `0003_unique_installment_seq.sql`, but `0003` is taken by `0003_invite_accept_fn.sql` from Session 3.

**Decision**: Create `0004_create_debt_fn.sql`. This file:
1. Notes the UNIQUE constraint already exists (idempotent ALTER TABLE with DO NOTHING pattern)
2. Defines the `create_debt_with_installments` PL/pgSQL function
3. Grants EXECUTE to service_role only

### Decision 5: due_day Capped at 28

Due day is DB-constrained to `CHECK (due_day BETWEEN 1 AND 28)`. Since every calendar month has ≥ 28 days (February ≥ 28 even in non-leap years), `make_date(year, month, dueDay)` will always succeed without clamping. No month-end edge case handling needed in the date computation.

## Rounding Invariant

The validation rule: `|total_amount_minor - installment_amount_minor × total_installments| ≤ total_installments`

This allows at most 1 minor unit per installment in rounding residual. Implementation:

```typescript
const diff = totalAmountMinor - installmentAmountMinor * BigInt(totalInstallments)
const absDiff = diff < 0n ? -diff : diff
if (absDiff > BigInt(totalInstallments)) {
  throw new Error(`Rounding invariant violated: residual ${diff} exceeds ±${totalInstallments}`)
}
```

For the PRD example: 591500 − 147875 × 4 = 591500 − 591500 = 0 ≤ 4 ✓

## Due Date Computation (TypeScript)

```typescript
// Sequence number is 1-indexed
function computeDueDate(startMonth: string, dueDay: number, sequenceNumber: number): string {
  const [y, m] = startMonth.split('-').map(Number)
  const zeroIndexedMonths = (y * 12 + m - 1) + (sequenceNumber - 1)
  const year = Math.floor(zeroIndexedMonths / 12)
  const month = (zeroIndexedMonths % 12) + 1  // back to 1-indexed
  return `${year}-${String(month).padStart(2, '0')}-${String(dueDay).padStart(2, '0')}`
}
```

Verification for Nov→Dec→Jan boundary:
- startMonth='2024-11', dueDay=25, n=1 → 2024-11-25 ✓
- n=2 → 2024-12-25 ✓  
- n=3 → 2025-01-25 ✓

## Files to Create

### 1. `supabase/migrations/0004_create_debt_fn.sql`

PL/pgSQL function for atomic debt + installment creation. Called exclusively by service_role.

**Function signature**: `create_debt_with_installments(p_debtor_id, p_currency, p_total_amount_minor, p_total_installments, p_installment_amount_minor, p_due_day, p_start_month, p_description, p_created_by) RETURNS uuid`

**Function body**:
1. INSERT into debts, capture RETURNING id into v_debt_id
2. FOR v_seq IN 1..p_total_installments LOOP
   - Compute due date: `v_start_date + make_interval(months => v_seq - 1)` then set day to p_due_day using `make_date`
   - INSERT INTO installments with ON CONFLICT (debt_id, sequence_number) DO NOTHING
3. RETURN v_debt_id

**Grant**: `GRANT EXECUTE ON FUNCTION public.create_debt_with_installments TO service_role`

**Type annotation in database.types.ts**: Add the function to the Functions section. The function returns `uuid` (string in TS). Since types are generated, add a comment-only note in the plan; actual implementation will use an `as string` cast at the trust boundary.

### 2. `src/lib/domain/installments/generateForDebt.ts`

**Exports**:
- `computeDueDate(startMonth: string, dueDay: number, sequenceNumber: number): string` — pure, no imports
- `computeInstallments(debtId: string, params: GenerateParams): InstallmentInsertRow[]` — pure, returns array
- `generateForDebt(adminClient: ReturnType<typeof createAdminClient>, debtId: string, params: GenerateParams): Promise<number>` — DB-aware, returns count of rows inserted (0 if already existed)

**Type `GenerateParams`**:
```typescript
type GenerateParams = {
  totalInstallments: number
  installmentAmountMinor: bigint
  dueDay: number
  startMonth: string
}
```

**Type `InstallmentInsertRow`** (internal, matches DB Insert type with number boundary):
```typescript
type InstallmentInsertRow = {
  debt_id: string
  sequence_number: number
  due_date: string
  amount_minor: number        // boundary: bigint→number for Supabase insert
  remaining_amount_minor: number  // boundary: bigint→number for Supabase insert
  status: 'pending'
}
```

**`generateForDebt` implementation**:
1. Call `computeInstallments` to get rows
2. `adminClient.from('installments').upsert(rows, { onConflict: 'debt_id,sequence_number', ignoreDuplicates: true })`
3. Return count of affected rows (use `.select()` chain to count, or just return rows.length as optimistic count)

Note: Supabase upsert with `ignoreDuplicates: true` is the JS-level equivalent of `ON CONFLICT DO NOTHING`.

**`server-only`**: Add `import 'server-only'` since generateForDebt uses the admin client.

### 4. `src/lib/domain/debts/createDebt.ts`

**Pattern**: `'use server'` + `action.schema(schema).action(handler)` (mirrors auth/actions.ts)

**Zod schema**:
```typescript
const createDebtSchema = z.object({
  debtor_id: z.string().uuid(),
  total_amount_minor: z.number().int().positive().max(Number.MAX_SAFE_INTEGER),
  total_installments: z.number().int().min(1).max(120),
  installment_amount_minor: z.number().int().positive().max(Number.MAX_SAFE_INTEGER),
  due_day: z.number().int().min(1).max(28),
  currency: z.enum(['CRC', 'USD']),
  start_month: z.string().regex(/^\d{4}-(0[1-9]|1[0-2])$/, 'Formato YYYY-MM requerido'),
  description: z.string().max(500).optional(),
})
```

**Handler**:
1. `const admin = await requireAdmin()`
2. Convert amounts to bigint: `const total = BigInt(total_amount_minor)`, `const installAmt = BigInt(installment_amount_minor)`
3. Assert rounding invariant (throw if violated — `next-safe-action` will convert to serverError)
4. Verify debtor exists and has role='debtor' via `adminClient.from('users').select('role').eq('id', debtor_id).single()`
5. Call `adminClient.rpc('create_debt_with_installments', { p_debtor_id, p_currency, p_total_amount_minor: Number(total), ... })`
6. If RPC error, throw descriptive error
7. Return `{ debtId: data as string }`

**Boundary notes**:
- `Number(total)` at DB insert — safe because DB CHECK constrains amounts > 0 and Zod constrains < MAX_SAFE_INTEGER
- `adminClient.rpc(...)` returns `Json` typed as `unknown`; cast result to `string` with boundary comment

### 10. `src/app/api/cron/generate-installments/route.ts`

GET handler (Vercel cron calls GET by default).

```typescript
import { NextRequest, NextResponse } from 'next/server'
import { createAdminClient } from '@/lib/supabase/admin'
import { generateForDebt } from '@/lib/domain/installments/generateForDebt'

export async function GET(request: NextRequest) {
  const authHeader = request.headers.get('Authorization')
  const cronSecret = process.env.CRON_SECRET
  
  if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }
  
  const admin = createAdminClient()
  
  // Find debts that have fewer installments than total_installments
  const { data: debts, error } = await admin
    .from('debts')
    .select('id, total_installments, installment_amount_minor, due_day, start_month')
    .eq('status', 'active')
  
  if (error) {
    console.error('[cron] failed to fetch debts:', error.message)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
  
  let backfilled = 0
  const errors: string[] = []
  
  for (const debt of debts ?? []) {
    const { count } = await admin
      .from('installments')
      .select('id', { count: 'exact', head: true })
      .eq('debt_id', debt.id)
    
    if ((count ?? 0) < debt.total_installments) {
      try {
        await generateForDebt(admin, debt.id, {
          totalInstallments: debt.total_installments,
          installmentAmountMinor: BigInt(debt.installment_amount_minor),
          dueDay: debt.due_day,
          startMonth: debt.start_month,
        })
        backfilled++
      } catch (err) {
        errors.push(`debt ${debt.id}: ${err instanceof Error ? err.message : String(err)}`)
      }
    }
  }
  
  console.log(`[cron] backfilled ${backfilled} debts, ${errors.length} errors`)
  return NextResponse.json({ backfilled, errors })
}
```

**`vercel.json`**: Note in handoff that `CRON_SECRET` env var must be set in Vercel dashboard and that the cron schedule (e.g., daily) needs to be added to `vercel.json`. Not implemented in this session — just the route stub.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| PL/pgSQL function type not in database.types.ts | Manually add function type; use `as string` cast at call site with boundary comment |
| `database.types.ts` uses `number` for monetary fields | Convert bigint→number at DB insert boundary with explicit comment; convert number→bigint at DB read boundary before any math |
| `<input type="month">` browser support | Safari on older iOS may not support; acceptable for admin-only tool; add fallback text input with regex validation |
| Cron route accessible without auth | CRON_SECRET header check before any logic; 401 if missing or wrong |
| Due date off-by-one in year boundary | computeDueDate unit tests cover this explicitly |
| Supabase upsert with ignoreDuplicates true | Test with mock that verifies the flag is passed; DB UNIQUE constraint is the actual enforcement layer |
| next-safe-action serverError swallows exception stack | Errors in handler become generic serverError strings; use descriptive error messages in throws |
| Zod on server action receives string from FormData | Form converts amounts to number before calling execute; Zod schema uses z.number().int() not z.string() |
