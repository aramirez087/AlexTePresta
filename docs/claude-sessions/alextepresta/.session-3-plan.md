## Key Design Decisions

### D1: Invite token persistence through OAuth round-trip
The invite token must survive the Google OAuth redirect. Options:
1. Encode token in OAuth `state` param — unreliable; Supabase may strip/transform it
2. Store token in a short-lived `HttpOnly` cookie before redirecting to OAuth

**Decision: HttpOnly cookie `pending_invite` (15-min TTL, SameSite=Lax).** Set it in the invite page server action before initiating OAuth, read it in the callback route after session is established, then immediately clear it. This is reliable, immune to OAuth state stripping, and the short TTL prevents stale tokens.

### D2: Atomicity of invite acceptance
The accept operation must atomically: validate token, upsert user, mark invite consumed. The Supabase JS client cannot issue raw `BEGIN/COMMIT` over PostgREST. 

**Decision: New migration `0003_invite_accept_fn.sql` with a `SECURITY DEFINER` Postgres function `accept_invite(p_token, p_user_id, p_email)`.** Called via the admin client's `.rpc()`. The function uses `SELECT ... FOR UPDATE` on the invite row to prevent concurrent double-acceptance, then performs upsert + consumed_at update in one transaction.

### D3: Role check in middleware vs. server helpers
Middleware needs to redirect non-admins away from `/admin/*`. Two options:
1. JWT custom claims (requires Supabase webhook/hook setup — deferred)
2. Query `public.users` for the role on each admin-route request

**Decision: Query `public.users` in middleware for `/admin/*` requests.** The admin is a single user; this is one extra DB round-trip only on admin paths. Acceptable for Phase 1. JWT claims optimization is noted for a future session.

### D4: Server action pattern
`next-safe-action` is already installed. Use it for `createInvite` to get input validation (Zod) + typed errors for free.

**Decision: Create `src/lib/safe-action.ts` with a thin action client, then use `next-safe-action` for `createInvite`.**

### D5: Typed Supabase clients
Session 2 deferred the `Database` generic type. Session 3 generates it first, then updates all three clients.

**Decision: Generate `src/lib/supabase/database.types.ts` at session start (Step 0). Update clients in the same commit.**

### D6: Invites RLS is admin-only
The `public.invites` RLS policy only allows admins. Token validation during invite acceptance (before the user is in `public.users`) must bypass RLS.

**Decision: All invite-related DB operations (token validation, user upsert, consumed_at update) use the admin client (service role) or the `accept_invite` SECURITY DEFINER function.**

### D7: Login page as hybrid Server + Client Component
The Google OAuth button triggers `supabase.auth.signInWithOAuth()`, which is a browser API. The page itself can be a Server Component; the button must be a Client Component.

**Decision: `src/app/login/page.tsx` is a Server Component. It renders `<SignInButton>` which is a separate Client Component in `src/app/login/_components/sign-in-button.tsx`.**

### Step 1 — `supabase/migrations/0003_invite_accept_fn.sql`

Create the atomic invite acceptance function. Full spec:

```sql
CREATE OR REPLACE FUNCTION public.accept_invite(
  p_token text,
  p_user_id uuid,
  p_email text
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_invite invites%ROWTYPE;
BEGIN
  -- Lock the invite row (prevent race conditions on concurrent acceptance)
  SELECT * INTO v_invite
  FROM public.invites
  WHERE token = p_token
  FOR UPDATE;

  IF NOT FOUND THEN
    RETURN jsonb_build_object('ok', false, 'error', 'token_not_found');
  END IF;

  IF v_invite.expires_at < now() THEN
    RETURN jsonb_build_object('ok', false, 'error', 'token_expired');
  END IF;

  IF v_invite.consumed_at IS NOT NULL THEN
    RETURN jsonb_build_object('ok', false, 'error', 'token_consumed');
  END IF;

  -- Invite email must match the Google account email exactly
  IF lower(v_invite.email) != lower(p_email) THEN
    RETURN jsonb_build_object('ok', false, 'error', 'email_mismatch');
  END IF;

  INSERT INTO public.users (id, email, role, invited_by)
  VALUES (p_user_id, p_email, 'debtor', v_invite.inviter_id)
  ON CONFLICT (id) DO UPDATE SET
    email = EXCLUDED.email,
    role = 'debtor',
    invited_by = EXCLUDED.invited_by;

  UPDATE public.invites
  SET consumed_at = now()
  WHERE id = v_invite.id;

  RETURN jsonb_build_object('ok', true);
END;
$$;

-- Grant execute only to authenticated and service_role (not anon)
REVOKE ALL ON FUNCTION public.accept_invite(text, uuid, text) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.accept_invite(text, uuid, text) TO service_role;
```

Note: Since the function is called exclusively via the admin client (service_role), only service_role needs execute permission.

### Step 3 — `src/lib/auth/session.ts`

Server-only helpers. Must import `server-only`.

```typescript
import 'server-only'
import { createClient } from '@/lib/supabase/server'
import { AuthRequiredError, ForbiddenError } from './errors'

// Returns the authenticated Supabase user or throws AuthRequiredError
export async function requireUser() {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) throw new AuthRequiredError()
  return user
}

// Returns the user only if they are an admin, otherwise throws
export async function requireAdmin() {
  const user = await requireUser()
  const supabase = await createClient()
  const { data } = await supabase
    .from('users')
    .select('role')
    .eq('id', user.id)
    .single()
  if (!data || data.role !== 'admin') throw new ForbiddenError()
  return user
}

// Returns role or null if not authenticated/not in public.users
export async function currentRole(): Promise<'admin' | 'debtor' | null> {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return null
  const { data } = await supabase
    .from('users')
    .select('role')
    .eq('id', user.id)
    .single()
  return (data?.role as 'admin' | 'debtor') ?? null
}
```

Note: `supabase.auth.getUser()` makes a network call to the Supabase Auth server (validates the JWT against the server, not just locally). This is the correct pattern for server components where security is required — `getSession()` only checks the local cookie.

### Step 5 — `src/lib/auth/actions.ts`

`createInvite` server action using next-safe-action + Zod:

```typescript
'use server'
import { z } from 'zod'
import { action } from '@/lib/safe-action'
import { createAdminClient } from '@/lib/supabase/admin'
import { requireAdmin } from '@/lib/auth/session'
import { randomBytes } from 'crypto'

export const createInvite = action
  .schema(z.object({ email: z.string().email() }))
  .action(async ({ parsedInput: { email } }) => {
    const inviter = await requireAdmin()
    const token = randomBytes(32).toString('hex')
    const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000).toISOString()

    const admin = createAdminClient()
    const { data, error } = await admin
      .from('invites')
      .insert({
        email,
        token,
        expires_at: expiresAt,
        inviter_id: inviter.id,
      })
      .select('token')
      .single()

    if (error) throw new Error(`Error al crear la invitación: ${error.message}`)

    return { token: data.token }
  })
```

Note: `randomBytes(32).toString('hex')` produces a 64-character hex string. This is a cryptographically secure 256-bit token, meeting the "32-byte" requirement.

`inviter.id` comes from the Supabase Auth `user.id`, not `public.users.id`. These are the same UUID because `public.users.id` mirrors `auth.users.id` (established in 0001_init.sql).

### Step 6 — `src/middleware.ts`

Pattern from `@supabase/ssr` docs. The critical rule: always return `supabaseResponse`, never create a secondary `NextResponse` without copying cookies.

```typescript
import { createServerClient } from '@supabase/ssr'
import { NextResponse, type NextRequest } from 'next/server'

export async function middleware(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request })

  const url = process.env.NEXT_PUBLIC_SUPABASE_URL!
  const key = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!

  const supabase = createServerClient(url, key, {
    cookies: {
      getAll() { return request.cookies.getAll() },
      setAll(cookiesToSet) {
        cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value))
        supabaseResponse = NextResponse.next({ request })
        cookiesToSet.forEach(({ name, value, options }) =>
          supabaseResponse.cookies.set(name, value, options)
        )
      },
    },
  })

  // Refresh session — must call getUser(), NOT getSession() here
  const { data: { user } } = await supabase.auth.getUser()

  const { pathname } = request.nextUrl

  const isProtected = pathname.startsWith('/app') || pathname.startsWith('/admin')
  if (!user && isProtected) {
    const loginUrl = request.nextUrl.clone()
    loginUrl.pathname = '/login'
    loginUrl.searchParams.set('next', pathname)
    return NextResponse.redirect(loginUrl)
  }

  if (user && pathname.startsWith('/admin')) {
    const { data } = await supabase
      .from('users')
      .select('role')
      .eq('id', user.id)
      .single()
    if (!data || data.role !== 'admin') {
      const appUrl = request.nextUrl.clone()
      appUrl.pathname = '/app'
      appUrl.search = ''
      return NextResponse.redirect(appUrl)
    }
  }

  return supabaseResponse
}

export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon\\.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
}
```

**Risk**: The `supabase.auth.getUser()` call in middleware adds ~50-100ms per request (network call to Supabase Auth). Mitigation: In a future session, switch to JWT custom claims so the role check is a local token parse. For Phase 1 traffic (1 admin, few debtors), this is acceptable.

**Risk**: On Vercel edge, if NEXT_PUBLIC_SUPABASE_URL or ANON_KEY are missing at runtime, the `!` assertion will cause a crash. Mitigation: The session start verification (`npm run build`) will fail if env vars are absent, catching this early.

### Step 8 — Auth callback route

**`src/app/auth/callback/route.ts`**:

```typescript
import { NextResponse, type NextRequest } from 'next/server'
import { createClient } from '@/lib/supabase/server'
import { createAdminClient } from '@/lib/supabase/admin'

export async function GET(request: NextRequest) {
  const { searchParams, origin } = new URL(request.url)
  const code = searchParams.get('code')
  const next = searchParams.get('next') ?? '/app'

  if (!code) {
    return NextResponse.redirect(`${origin}/login?error=missing_code`)
  }

  const supabase = await createClient()
  const { error } = await supabase.auth.exchangeCodeForSession(code)
  if (error) {
    return NextResponse.redirect(`${origin}/login?error=auth_failed`)
  }

  // Check for pending invite token (set as cookie by /invite/[token])
  const pendingToken = request.cookies.get('pending_invite')?.value

  if (pendingToken) {
    const { data: { user } } = await supabase.auth.getUser()
    if (user?.email) {
      const admin = createAdminClient()
      const { data: result } = await admin.rpc('accept_invite', {
        p_token: pendingToken,
        p_user_id: user.id,
        p_email: user.email,
      })

      const response = NextResponse.redirect(`${origin}${next}`)
      // Clear the pending invite cookie regardless of result
      response.cookies.delete('pending_invite')

      if (!result?.ok) {
        const errPath = `/invite/error?reason=${result?.error ?? 'unknown'}`
        const errResponse = NextResponse.redirect(`${origin}${errPath}`)
        errResponse.cookies.delete('pending_invite')
        return errResponse
      }

      return response
    }
  }

  return NextResponse.redirect(`${origin}${next}`)
}
```

Note: After `exchangeCodeForSession`, the session is in cookies. `supabase.auth.getUser()` validates against the Auth server. We don't need to create a new client — the same instance has the freshly-set session cookie.

### Step 9 — Invite acceptance page

**`src/app/invite/[token]/page.tsx`** — Server Component:

```typescript
import { notFound } from 'next/navigation'
import { createAdminClient } from '@/lib/supabase/admin'
import { createClient } from '@/lib/supabase/server'
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'

// Spanish error messages keyed by error code
const ERROR_MESSAGES: Record<string, string> = {
  token_not_found: 'El enlace de invitación no es válido.',
  token_expired: 'El enlace de invitación ha expirado. Solicite uno nuevo al administrador.',
  token_consumed: 'Este enlace de invitación ya fue utilizado.',
  email_mismatch: 'Su cuenta de Google no coincide con el correo de la invitación.',
}

interface Props {
  params: Promise<{ token: string }>
}

export default async function InvitePage({ params }: Props) {
  const { token } = await params

  // Validate token via admin client (invites table is admin-only RLS)
  const admin = createAdminClient()
  const { data: invite, error } = await admin
    .from('invites')
    .select('email, expires_at, consumed_at')
    .eq('token', token)
    .single()

  if (error || !invite) {
    return <ErrorPage message={ERROR_MESSAGES.token_not_found} />
  }

  if (new Date(invite.expires_at) < new Date()) {
    return <ErrorPage message={ERROR_MESSAGES.token_expired} />
  }

  if (invite.consumed_at) {
    return <ErrorPage message={ERROR_MESSAGES.token_consumed} />
  }

  // Check if user is already signed in
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (user) {
    // Already authenticated — attempt direct acceptance
    const { data: result } = await admin.rpc('accept_invite', {
      p_token: token,
      p_user_id: user.id,
      p_email: user.email ?? '',
    })

    if (!result?.ok) {
      return <ErrorPage message={ERROR_MESSAGES[result?.error] ?? 'Error desconocido.'} />
    }

    redirect('/app')
  }

  // Not signed in — set cookie and redirect to login (which triggers OAuth)
  const cookieStore = await cookies()
  cookieStore.set('pending_invite', token, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax',
    maxAge: 15 * 60, // 15 minutes
    path: '/',
  })

  redirect('/login?next=/app')
}

function ErrorPage({ message }: { message: string }) {
  return (
    <main className="...">
      <h1 className="...">Invitación inválida</h1>
      <p className="...">{message}</p>
      <p className="...">Contacte al administrador para obtener un nuevo enlace.</p>
    </main>
  )
}
```

**Risk**: Server Component cannot set cookies directly in Next.js 15 — `cookies().set()` throws if called in a Server Component that is not a Server Action. However, Next.js 15 does allow `cookies().set()` in Server Components when you `await cookies()` from `next/headers`. This IS supported per Next.js 15 docs. Verify this behavior during implementation.

**Alternative**: If `cookies().set()` does not work in a Server Component, move the cookie-setting + redirect logic into a Server Action (form with hidden input), and render a form that auto-submits. This is more complex but fully supported.

**Preferred mitigation**: Test `cookies().set()` behavior in a simple page first. If it fails, use a Server Action pattern:
```typescript
// Fallback: use a Server Action
async function initiateOAuth(token: string) {
  'use server'
  const cookieStore = await cookies()
  cookieStore.set('pending_invite', token, { httpOnly: true, maxAge: 900, path: '/' })
  redirect('/login?next=/app')
}
```
Then render `<form action={initiateOAuth.bind(null, token)}><button type="submit">Continuar con Google</button></form>`.

## Risks and Mitigations

### R1: `cookies().set()` in Server Component
**Risk**: Next.js 15 may throw when setting cookies in a Server Component page (vs. a Server Action).
**Mitigation**: Test this first. If it fails, refactor the invite page to render a `<form>` that calls a Server Action to set the cookie + redirect. The Server Action approach is always supported.

### R2: `accept_invite` RPC called twice if user re-visits
**Risk**: If Google OAuth completes but redirect to `/auth/callback` fails (network error), the user's browser may retry. The second call to `accept_invite` would find `consumed_at` already set and return `token_consumed`.
**Mitigation**: The function returns `token_consumed` error. The callback route should treat `token_consumed` as a success (the invite was accepted, perhaps in a previous attempt) and redirect to `/app` rather than showing an error. Add this special case to the callback handler.

### R3: Invite link can be brute-forced
**Risk**: 64-char hex tokens are 256-bit, so brute force is infeasible. However, tokens exposed in URL paths are visible in server logs, Referer headers, browser history.
**Mitigation**: Token is invalidated immediately on use (`consumed_at`). 7-day TTL. Tokens are not returned in list views (only used for the copy URL at creation time). Acceptable for Phase 1.

### R4: Session cookies not propagated in middleware redirect
**Risk**: The Supabase `@supabase/ssr` pattern requires that `supabaseResponse` (with refreshed session cookies) is always returned, even on redirects.
**Mitigation**: When redirecting in middleware, create the redirect response first, then copy session cookies from `supabaseResponse` to the redirect response, then return the redirect response. Follow the exact pattern from Supabase docs.

Revised redirect pattern for middleware:
```typescript
const redirectResponse = NextResponse.redirect(loginUrl)
// Copy session cookies from supabaseResponse to redirectResponse
supabaseResponse.cookies.getAll().forEach(cookie =>
  redirectResponse.cookies.set(cookie.name, cookie.value, { ...cookie })
)
return redirectResponse
```

### R5: `next-safe-action` version compatibility
**Risk**: `next-safe-action@7.x` API may differ from assumed patterns.
**Mitigation**: Check actual installed version (`"next-safe-action": "^7.10.8"`). Use `createSafeActionClient` from v7 docs. The `.schema().action()` chain is the v7 API.

### R6: `useAction` hook requires React client context
**Risk**: `next-safe-action/hooks` uses React hooks and must be in a Client Component.
**Mitigation**: Already planned — `CreateInviteForm` is a Client Component.

## Edge Cases to Handle

1. **User visits `/invite/{token}` while already signed in with the WRONG Google account**: Show email_mismatch error with message indicating which email is expected. Do NOT reveal the invite email in full — show only the domain (privacy).

2. **Admin signs in via invite link by accident**: The `accept_invite` function will try to set their role to 'debtor'. Guard against this: before calling `accept_invite`, check if the authenticated user's role is already 'admin', and if so show an error message.

3. **Token in cookie expires before OAuth completes**: The cookie TTL is 15 minutes, and Google OAuth typically completes in under 2 minutes. If the user takes longer (abandoned flow), the cookie will be absent in the callback. In this case, the callback route simply completes the session without invite acceptance and redirects to `/app` — where RLS will block any `/app` content until the user is in `public.users`. This is acceptable; the user can re-visit the invite link.

4. **`/auth/callback` called without a `code` param** (e.g., direct URL visit): Handled by the `if (!code)` guard — redirects to `/login?error=missing_code`.

5. **Invite acceptance race condition**: Two browser tabs simultaneously accepting the same invite. The `SELECT ... FOR UPDATE` in `accept_invite` serializes concurrent attempts — the second will see `consumed_at` set and return `token_consumed`.
