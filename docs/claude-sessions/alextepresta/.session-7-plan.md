## 3. Key Design Decisions

### 3.1 Migration 0006: Phase 2 apply_payment function

The entire payment logic lives in PL/pgSQL (`apply_payment` RPC). The TypeScript wrapper (`applyPayment.ts`) is already correct — it already declares `target_type: 'installment' | 'interest_debt'` in `ApplicationResult`. No TypeScript changes needed in `applyPayment.ts` itself.

The Phase 2 logic modifies the PL/pgSQL function to handle two new cases in the installment FIFO loop:
1. **Partial payment (payment exhausted before installment):** When `v_new_remaining > 0` after applying, instead of leaving the installment at partial remaining, create an `interest_debts` row and mark installment `status='converted'`.
2. **Interest debt FIFO:** After the installment loop, if payment still has remaining funds, apply FIFO to `interest_debts WHERE is_simulated=false AND status='active' ORDER BY created_at ASC`.

**Settings snapshot:** Read `default_annual_rate` from `settings` once at function start (before the loop), not per-iteration. The value is stored as a JSONB string `'"0.24"'`; extract with `value #>> '{}'` (no `trim()` needed, gets `'0.24'` directly).

### 3.2 accrueOne.ts: Rounding semantics

The task says: "47875 * (1 + 0.24/12) = 48832.5 → rounds to 48833"

**Verification:** The accrued amount is `47875 * 0.02 = 957.5`. Using half-even rounding on the **accrued amount** (not the closing balance): 957 is odd, 958 is even → 957.5 rounds to **958**. Then closing = 47875 + 958 = **48833** ✓.

The critical implementation detail: **round the accrued portion, not the closing balance**. Then add integer accrued to integer opening:

```typescript
const accruedDec = new Decimal(opening.toString()).mul(monthlyRate)
const accrued_minor = BigInt(accruedDec.toDecimalPlaces(0, Decimal.ROUND_HALF_EVEN).toFixed(0))
const closing_minor = opening + accrued_minor
```

This is correct because:
- `Decimal.ROUND_HALF_EVEN` on 957.5 → 958 (958 is even)
- closing_minor = integer arithmetic (no precision loss)

### 3.3 runMonthlyAccrual.ts: Idempotency

The `interest_accruals` table has `UNIQUE (interest_debt_id, period, mode)`. Idempotency is achieved by:
1. Attempting to `insert` with `ignoreDuplicates: true` via Supabase upsert.
2. Only updating `current_balance_minor` on `interest_debts` if the insert was a new record (not a duplicate).

Use a two-step approach: check existence first, then insert+update. This is clearer than inferring from upsert return metadata:

```typescript
const { data: existing } = await adminClient
  .from('interest_accruals')
  .select('id')
  .eq('interest_debt_id', debt.id)
  .eq('period', period)
  .eq('mode', 'real')
  .maybeSingle()

if (existing) { skipped++; continue }
// then insert accrual + update balance
```

Each interest_debt uses its **snapshot rate** (stored in `interest_debts.interest_rate`), not the current `settings` default. This ensures rate changes only affect future conversions.

## 4. Exact Order of Operations

### Step 1: Migration `0006_phase2_apply_payment_fn.sql`

Full Phase 2 `CREATE OR REPLACE FUNCTION public.apply_payment`:

```sql
-- Migration 0006: Phase 2 — partial payment converts installment remainder to interest_debt,
-- then applies remaining payment FIFO to active interest_debts.

CREATE OR REPLACE FUNCTION public.apply_payment(p_payment_id uuid)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_payment         payments%ROWTYPE;
  v_remaining       bigint;
  v_installment     installments%ROWTYPE;
  v_interest_debt   interest_debts%ROWTYPE;
  v_to_apply        bigint;
  v_new_remaining   bigint;
  v_applications    jsonb := '[]'::jsonb;
  v_default_rate    text;
BEGIN
  SELECT * INTO v_payment FROM payments WHERE id = p_payment_id FOR UPDATE;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'PaymentNotFoundError: payment % not found', p_payment_id;
  END IF;
  IF v_payment.status <> 'pending' THEN
    RAISE EXCEPTION 'PaymentAlreadyAppliedError: payment % is not pending', p_payment_id;
  END IF;

  v_remaining := v_payment.amount_minor;

  -- Read rate snapshot once (jsonb string → text via #>> '{}')
  SELECT value #>> '{}' INTO v_default_rate FROM settings WHERE key = 'default_annual_rate';

  -- FIFO: pending installments for this debtor's debts in matching currency
  FOR v_installment IN
    SELECT i.*
    FROM installments i
    JOIN debts d ON d.id = i.debt_id
    WHERE d.debtor_id = v_payment.debtor_id
      AND d.currency  = v_payment.currency
      AND i.status    = 'pending'
    ORDER BY i.due_date ASC, i.sequence_number ASC
    FOR UPDATE OF i
  LOOP
    EXIT WHEN v_remaining = 0;

    v_to_apply      := LEAST(v_remaining, v_installment.remaining_amount_minor);
    v_new_remaining := v_installment.remaining_amount_minor - v_to_apply;

    INSERT INTO payment_applications
      (payment_id, target_type, target_id, applied_amount_minor)
    VALUES
      (p_payment_id, 'installment', v_installment.id, v_to_apply);

    v_applications := v_applications || jsonb_build_array(
      jsonb_build_object(
        'target_id',            v_installment.id,
        'target_type',          'installment',
        'applied_amount_minor', v_to_apply
      )
    );

    v_remaining := v_remaining - v_to_apply;

    IF v_new_remaining > 0 THEN
      -- Phase 2: payment exhausted before covering installment — convert remainder
      INSERT INTO interest_debts
        (debt_id, source_installment_id, principal_minor, current_balance_minor,
         interest_rate, is_simulated, status)
      VALUES
        (v_installment.debt_id, v_installment.id, v_new_remaining, v_new_remaining,
         v_default_rate, false, 'active');

      UPDATE installments
      SET remaining_amount_minor = 0,
          status                 = 'converted'
      WHERE id = v_installment.id;
    ELSE
      UPDATE installments
      SET remaining_amount_minor = 0,
          status                 = 'paid'
      WHERE id = v_installment.id;
    END IF;
  END LOOP;

  -- Phase 2: apply remaining payment FIFO to active non-simulated interest_debts
  IF v_remaining > 0 THEN
    FOR v_interest_debt IN
      SELECT id.*
      FROM interest_debts id
      JOIN debts d ON d.id = id.debt_id
      WHERE d.debtor_id    = v_payment.debtor_id
        AND d.currency     = v_payment.currency
        AND id.is_simulated = false
        AND id.status       = 'active'
      ORDER BY id.created_at ASC
      FOR UPDATE OF id
    LOOP
      EXIT WHEN v_remaining = 0;

      v_to_apply      := LEAST(v_remaining, v_interest_debt.current_balance_minor);
      v_new_remaining := v_interest_debt.current_balance_minor - v_to_apply;

      INSERT INTO payment_applications
        (payment_id, target_type, target_id, applied_amount_minor)
      VALUES
        (p_payment_id, 'interest_debt', v_interest_debt.id, v_to_apply);

      v_applications := v_applications || jsonb_build_array(
        jsonb_build_object(
          'target_id',            v_interest_debt.id,
          'target_type',          'interest_debt',
          'applied_amount_minor', v_to_apply
        )
      );

      v_remaining := v_remaining - v_to_apply;

      UPDATE interest_debts
      SET current_balance_minor = v_new_remaining,
          status = CASE WHEN v_new_remaining = 0 THEN 'settled' ELSE status END
      WHERE id = v_interest_debt.id;
    END LOOP;
  END IF;

  IF v_remaining > 0 THEN
    RAISE EXCEPTION 'PaymentExcessError: payment % has % minor units unallocated',
      p_payment_id, v_remaining;
  END IF;

  UPDATE payments
  SET status = 'approved', applied_at = now()
  WHERE id = p_payment_id;

  RETURN jsonb_build_object(
    'applications', v_applications,
    'leftover_minor', 0
  );
END;
$$;

REVOKE ALL ON FUNCTION public.apply_payment(uuid) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.apply_payment(uuid) TO service_role;
```

**SQL alias issue:** In the FOR loop for interest_debts, `id.*` conflicts with the `id` column. Use a table alias. Rename `id` alias to `idb` or spell out the FROM clause as `FROM interest_debts idb JOIN debts d ON d.id = idb.debt_id` and `v_interest_debt` ROWTYPE from `interest_debts`. The `ROWTYPE` declared as `interest_debts%ROWTYPE` is fine.

**Corrected interest_debts loop (no alias conflict):**
```sql
FOR v_interest_debt IN
  SELECT idb.*
  FROM interest_debts idb
  JOIN debts d ON d.id = idb.debt_id
  WHERE d.debtor_id     = v_payment.debtor_id
    AND d.currency      = v_payment.currency
    AND idb.is_simulated = false
    AND idb.status       = 'active'
  ORDER BY idb.created_at ASC
  FOR UPDATE OF idb
LOOP
```

### Step 2: `src/lib/domain/interest/accrueOne.ts`

```typescript
import Decimal from 'decimal.js'

export function accrueOne(
  opening: bigint,
  monthlyRate: Decimal,
): { accrued_minor: bigint; closing_minor: bigint } {
  if (opening === 0n) return { accrued_minor: 0n, closing_minor: 0n }

  const accruedDec = new Decimal(opening.toString()).mul(monthlyRate)
  const accrued_minor = BigInt(
    accruedDec.toDecimalPlaces(0, Decimal.ROUND_HALF_EVEN).toFixed(0),
  )
  return { accrued_minor, closing_minor: opening + accrued_minor }
}
```

No `'server-only'` import — this is a pure function, testable in any environment.

### Step 3: `src/lib/domain/interest/__tests__/accrueOne.test.ts`

Test cases:
1. **PRD §7 canonical:** `opening=47875n, rate=0.02` → `accrued=958n, closing=48833n`
   - 47875 × 0.02 = 957.5 → ROUND_HALF_EVEN → 958 (958 is even), closing = 48833
2. **Zero principal:** `opening=0n` → both 0n
3. **Exact integers (no rounding):** `opening=100000n, rate=0.02` → `accrued=2000n, closing=102000n`
4. **Half-even tie toward even:** pick a value where accrued is exactly x.5 and x is even → rounds down
   - E.g., `opening=50000n, rate=0.02` → 1000.0 (no tie, exact), try `opening=75000n, rate=0.02` → 1500.0 (exact)
   - Use `opening=25000n, rate=0.02` → 500.0 (exact)
   - For a down-tie: `opening=47875n` already tests up-tie. For down-tie: need accrued = x.5 where x is even.
   - x must be even: e.g., 2.5 → opening * rate = 2.5. With rate=0.02: opening=125. 125 * 0.02 = 2.5, round to 2 (even). closing=127.
   - Test: `opening=125n, rate=0.02` → `accrued=2n, closing=127n`
5. **Large value:** `opening=999_999_999_999n` → no crash, BigInt arithmetic correct
6. **Two-period compounding:** period 1 from 47875n → closing=48833n; period 2 from 48833n → 48833 * 0.02 = 976.66 → rounds to 977 → closing=49810n

### Step 4: `src/lib/domain/interest/runMonthlyAccrual.ts`

```typescript
import 'server-only'
import Decimal from 'decimal.js'
import type { createAdminClient } from '@/lib/supabase/admin'
import { accrueOne } from './accrueOne'

export type AccrualSummary = {
  processed: number
  skipped: number
  errors: string[]
}

export async function runMonthlyAccrual(
  adminClient: ReturnType<typeof createAdminClient>,
  period: string,
): Promise<AccrualSummary> {
  const { data: debts } = await adminClient
    .from('interest_debts')
    .select('id, current_balance_minor, interest_rate')
    .eq('is_simulated', false)
    .eq('status', 'active')

  let processed = 0
  let skipped = 0
  const errors: string[] = []

  for (const debt of debts ?? []) {
    try {
      const { data: existing } = await adminClient
        .from('interest_accruals')
        .select('id')
        .eq('interest_debt_id', debt.id)
        .eq('period', period)
        .eq('mode', 'real')
        .maybeSingle()

      if (existing) {
        skipped++
        continue
      }

      const annualRate = new Decimal(debt.interest_rate)
      const monthlyRate = annualRate.div(12)
      const opening = BigInt(debt.current_balance_minor)
      const { accrued_minor, closing_minor } = accrueOne(opening, monthlyRate)

      const { error: insertError } = await adminClient
        .from('interest_accruals')
        .insert({
          interest_debt_id:      debt.id,
          period,
          opening_balance_minor: Number(opening),
          accrued_amount_minor:  Number(accrued_minor),
          closing_balance_minor: Number(closing_minor),
          mode:                  'real',
        })

      if (insertError) throw new Error(insertError.message)

      const { error: updateError } = await adminClient
        .from('interest_debts')
        .update({ current_balance_minor: Number(closing_minor) })
        .eq('id', debt.id)

      if (updateError) throw new Error(updateError.message)

      processed++
    } catch (err) {
      errors.push(`${debt.id}: ${err instanceof Error ? err.message : String(err)}`)
    }
  }

  return { processed, skipped, errors }
}
```

**Bigint → number boundary:** `current_balance_minor` is stored as bigint in DB but Supabase returns `number`. The `Number(closing_minor)` conversion is safe per domain invariant (interest debt balances < MAX_SAFE_INTEGER). Add an inline comment at each boundary.

**Why per-debt error handling (try/catch in loop):** One failed debt should not abort the rest. Errors collected and returned to cron route for logging.

import { NextRequest, NextResponse } from 'next/server'
import { createAdminClient } from '@/lib/supabase/admin'
import { runMonthlyAccrual } from '@/lib/domain/interest/runMonthlyAccrual'

export async function GET(request: NextRequest) {
  const authHeader = request.headers.get('Authorization')
  const cronSecret = process.env.CRON_SECRET

  if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  const now = new Date()
  // Compute period in Costa Rica timezone ('YYYY-MM-DD' via en-CA → slice first 7 chars)
  const period = new Intl.DateTimeFormat('en-CA', {
    timeZone: 'America/Costa_Rica',
    year: 'numeric',
    month: '2-digit',
  }).format(now).slice(0, 7)

  const admin = createAdminClient()

  try {
    const summary = await runMonthlyAccrual(admin, period)
    console.log(`[cron/monthly-accrual] period=${period}`, summary)
    return NextResponse.json({ period, ...summary })
  } catch (err) {
    const message = err instanceof Error ? err.message : String(err)
    console.error('[cron/monthly-accrual] fatal:', message)
    return NextResponse.json({ error: message }, { status: 500 })
  }
}

```typescript
'use server'
import { createAdminClient } from '@/lib/supabase/admin'
import { requireAdmin } from '@/lib/auth/session'
import { revalidatePath } from 'next/cache'

export async function updateDefaultRate(formData: FormData): Promise<{ error?: string }> {
  const admin = await requireAdmin()
  const raw = formData.get('rate')
  if (typeof raw !== 'string') return { error: 'Valor inválido' }

  const trimmed = raw.trim()
  const parsed = parseFloat(trimmed)
  if (isNaN(parsed) || parsed <= 0 || parsed >= 1) {
    return { error: 'La tasa debe ser un número entre 0 y 1 (ej: 0.24 para 24%)' }
  }

  const adminClient = createAdminClient()
  const { error } = await adminClient
    .from('settings')
    .upsert({
      key:        'default_annual_rate',
      value:      JSON.stringify(trimmed),
      updated_by: admin.id,
      updated_at: new Date().toISOString(),
    })

  if (error) return { error: error.message }

  revalidatePath('/admin/settings')
  return {}
}
```

Return type `{ error?: string }` allows the page to show validation messages without using `next-safe-action` (simpler for a one-field settings form).

## 5. Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| SQL alias conflict: `id` column vs `id` table alias in interest_debts loop | Use explicit alias `idb` throughout the loop query |
| `FOR UPDATE OF idb` syntax in joined query | PostgreSQL allows `FOR UPDATE OF <alias>` in joined FOR loops; verify syntax in migration comment |
| `settings.value` is JSONB string `'"0.24"'` | Use `value #>> '{}'` in SQL to extract text without quotes |
| bigint→number at DB boundary in runMonthlyAccrual | Add invariant comments; domain amounts < MAX_SAFE_INTEGER per operator rules |
| Cron period drift at month boundary (request arrives just after midnight CR time) | Using CR timezone explicitly eliminates UTC offset issues |
| `updateDefaultRate` action — `data?.value` type from Supabase is `Json` not `string` | Stringify with `String(data.value).replace(/"/g, '')` or check with typeof |
| Existing `applyPayment` tests for `PaymentExcessError` may need review | Existing tests remain valid; they mock the RPC error response, not the DB behavior |
| `interest_accruals UNIQUE` includes `mode` — runMonthlyAccrual uses `mode='real'` | Filter `.eq('mode', 'real')` in idempotency check; correct and aligned with schema |
| `getDebtorOverview` mock in tests: `from()` mock now needs to handle 3 tables | Update `makeAdminClient` helper to handle `interest_debts` table case |
