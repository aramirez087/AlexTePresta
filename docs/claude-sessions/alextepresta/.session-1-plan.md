### Step 1 — `docs/architecture/charter.md`

**Sections:**
- **Vision**: Personal loan tracking for family/friends; admin manages lenders/borrowers; debtors see their own state. Access by invitation only.
- **Baseline**: Repo is empty (no application code) as of session 1.
- **Target users**: One admin (Alexander Ramirez), multiple debtors (non-technical family members). Spanish UI (es-CR locale).
- **Phase 1 (MVP)**: Google login, invitation flow, zero-rate debts, installment schedule generation, debtor payment submission, admin approval, FIFO application to installments, audit trail (`payment_applications`), account-statement timeline view.
- **Phase 2**: Partial-payment conversion → `interest_debts` at compound rate; monthly accrual cron; admin settings for default rate.
- **Phase 3**: Simulation mode — parallel simulated interest track; scenario projector; amber UI banner; no DB side-effects from toggle.
- **Out of scope (all phases)**: Multi-admin, currency conversion, recurring payments, PDF exports, push notifications, external integrations.
- **Glossary** (must define):
  - `Debt` — a zero-rate installment obligation in a single currency, created by admin for a debtor.
  - `Installment` — one scheduled payment slice of a `Debt`, identified by `sequence_number`, with `due_date`, `amount_minor`, `remaining_amount_minor`, and `status ∈ {pending, paid, converted, overdue}`.
  - `InterestDebt` — a compound-interest sub-obligation created when a payment partially covers an installment. Carries a snapshot `interest_rate`, `principal_minor`, and running `current_balance_minor`.
  - `Payment` — money received (or submitted by debtor), `status ∈ {pending, approved, rejected}`. Currency must match the target debts.
  - `PaymentApplication` — immutable audit record linking a `Payment` to an `Installment` or `InterestDebt`, recording `applied_amount_minor`. Never deleted or updated.
  - `Accrual` — monthly record of compound interest growth on an `InterestDebt`, keyed `(interest_debt_id, period, mode)`. Idempotent.

### Step 3 — `docs/architecture/payment-pipeline.md`

**Phase 1 — FIFO (no interest):**

Numbered pseudocode:
1. Caller must hold admin role or be the debtor owner. Validate via `requireAdmin()` or `requireUser()`.
2. `BEGIN TRANSACTION`
3. `SELECT * FROM payments WHERE id = $id FOR UPDATE` — lock row; assert `status = 'pending'`.
4. Resolve `debtor_id` from the payment row.
5. `SELECT i.* FROM installments i JOIN debts d ON d.id = i.debt_id WHERE d.debtor_id = $debtor_id AND d.currency = $payment.currency AND i.status = 'pending' ORDER BY i.due_date ASC, i.sequence_number ASC FOR UPDATE` — locks all candidate rows to prevent concurrent double-application.
6. Initialize `remaining_minor = payment.amount_minor`.
7. For each installment (FIFO order):
   a. If `remaining_minor = 0`, break.
   b. `to_apply = MIN(remaining_minor, installment.remaining_amount_minor)`
   c. `INSERT INTO payment_applications (payment_id, target_type='installment', target_id=installment.id, applied_amount_minor=to_apply)`
   d. `UPDATE installments SET remaining_amount_minor -= to_apply WHERE id = installment.id`
   e. If `installment.remaining_amount_minor = 0`: `UPDATE installments SET status = 'paid'`
   f. `remaining_minor -= to_apply`
8. If `remaining_minor > 0`: **ROLLBACK** and raise `PaymentExcessError` (Phase 1 rejects excess).
9. `UPDATE payments SET status = 'approved', applied_at = now() WHERE id = $id`
10. `COMMIT`
11. Return `{ applications: [...], leftover_minor: 0 }`.

**Transaction boundary**: Steps 2–10 form one atomic transaction. The `SELECT ... FOR UPDATE` in step 3 and step 5 prevents concurrent `applyPayment` calls for the same debtor from interleaving.

**Phase 2 — Partial-payment conversion:**

Steps 1–6 identical to Phase 1. Step 7 changes:

7. For each installment (FIFO):
   a. If `remaining_minor = 0`, break.
   b. If `remaining_minor >= installment.remaining_amount_minor` (full or over-covers):
      - `to_apply = installment.remaining_amount_minor`
      - Insert `payment_applications` row, set `remaining_amount_minor = 0`, `status = 'paid'`
      - `remaining_minor -= to_apply`
   c. Else (`remaining_minor < installment.remaining_amount_minor`) — **partial**:
      - `to_apply = remaining_minor`
      - Insert `payment_applications` (target_type='installment')
      - `shortfall = installment.remaining_amount_minor - to_apply`
      - Read `rate = settings['default_annual_rate']` (snapshot)
      - `INSERT INTO interest_debts (debt_id=debt.id, source_installment_id=installment.id, principal_minor=shortfall, current_balance_minor=shortfall, interest_rate=rate, is_simulated=false)`
      - `UPDATE installments SET status='converted', remaining_amount_minor=0`
      - `remaining_minor = 0`; break.

8. If `remaining_minor > 0` after installments loop, apply FIFO to `interest_debts`:
   - `SELECT * FROM interest_debts WHERE debt_id IN (debtor's debts) AND is_simulated=false AND status='active' ORDER BY created_at ASC FOR UPDATE`
   - For each: `to_apply = MIN(remaining, current_balance_minor)`, insert `payment_applications (target_type='interest_debt')`, update `current_balance_minor`, if 0 → `status='settled'`.

9. If `remaining_minor > 0` after both loops: ROLLBACK and raise `PaymentExcessError`.
10. `UPDATE payments SET status='approved', applied_at=now()`
11. `COMMIT`

**Critical note**: `payment_applications` rows are never updated or deleted. They are the event source for reconstructing any account state.

### Step 4 — `docs/architecture/interest-model.md`

**Monthly compound formula:**
```
monthly_rate = Decimal(annual_rate) / Decimal(12)
accrued_minor = ROUND_HALF_EVEN(Decimal(opening_balance_minor) * monthly_rate)
closing_minor = opening_balance_minor + accrued_minor
```
- Arithmetic uses `decimal.js` with sufficient precision (recommend `Decimal.set({ precision: 40 })`).
- Final cast to `bigint` uses half-even (banker's rounding) to avoid systematic bias over many accruals.
- `annual_rate` is persisted as a `text` column (decimal string) on `interest_debts` at creation time — this is a **rate snapshot**. Admin changes to `settings.default_annual_rate` do not retroactively alter existing debts.

**Real vs simulated mode:**
- `is_simulated = false` rows represent actual outstanding obligations.
- `is_simulated = true` rows are informational only; never factored into FIFO payment allocation.
- `interest_accruals.mode` column distinguishes which track an accrual record belongs to.

**Idempotency:**
- `UNIQUE (interest_debt_id, period, mode)` constraint on `interest_accruals`.
- The accrual cron uses `INSERT ... ON CONFLICT (interest_debt_id, period, mode) DO NOTHING`.
- Second invocation for the same `(period, mode)` is a strict no-op with no side effects.
- `period` format is 'YYYY-MM' (e.g., '2026-04'). The cron derives `period` from the wall-clock date at execution time.

**Rounding example (PRD §7):**
- `opening = 47875` (CRC minor = céntimos)
- `annual_rate = 0.24`, `monthly_rate = 0.02`
- `accrued = ROUND_HALF_EVEN(47875 × 0.02) = ROUND_HALF_EVEN(957.5) = 958` (half-even rounds 0.5 to nearest even → 958)
- `closing = 48833`

This exact assertion must appear as a unit test in Session 7.

### Step 5 — `docs/architecture/ux-partial-payment.md`

**Context**: PRD §13 identifies partial payments as the riskiest UX surface because a non-technical debtor must understand that paying less than a full installment creates a new compound-interest debt. The UI must make this **unmistakable before the user confirms**.

**Flow:**

1. **Debtor entry screen** (`/app/pay`): amount input, currency selector (disabled if debtor has only one currency). As debtor types, client-side preview computes projected allocation against pending installments (read-only calc, no server writes).

   > **Aviso**: El monto ingresado no cubre completamente ninguna cuota. Si continúas, la diferencia de ₡{shortfall} quedará registrada como **deuda con interés compuesto** al {rate}% anual.

   - Title: `¿Confirmar pago parcial?`
   - Body: `Vas a registrar un pago de ₡{amount}. La cuota #{N} por ₡{installment_amount} no quedará totalmente cubierta. La diferencia de ₡{shortfall} se convertirá en una deuda con interés compuesto del {rate}% anual a partir de hoy.`
   - Primary button: `Confirmar pago parcial` (red/destructive styling)
   - Cancel button: `Cancelar` (neutral)

4. **Visual cues required**:
   - Partial-payment warning banner: `bg-yellow-50 border-yellow-400 text-yellow-800`
   - Interest conversion notice: `bg-red-50 border-red-400 text-red-800`
   - Confirmation modal primary action: `bg-red-600 text-white`

#### `docs/adr/0002-money-representation.md`
- **Status**: Accepted
- **Decision**: All monetary amounts stored as `bigint` (PostgreSQL `bigint`, TypeScript `bigint`) in minor units (céntimos for CRC, cents for USD). `decimal.js` used for intermediate arithmetic. Results rounded half-even to integer minor units before persistence.
- **Context**: Floating-point arithmetic produces rounding errors that compound over many accrual periods. Using `number` for money is a well-known source of financial bugs.
- **Consequences**:
  - Pro: Exact integer arithmetic for storage; no silent rounding.
  - Pro: `decimal.js` half-even rounding matches GAAP/accounting standard.
  - Con: `bigint` cannot be serialized to JSON natively — must convert to `string` at API boundaries (trust boundary, documented with `// boundary: JSON serialization`).
  - Con: All UI must divide minor units by the currency's minor-unit factor (100 for both CRC and USD) using `Intl.NumberFormat`.

#### `docs/adr/0003-authorization-model.md`
- **Status**: Accepted
- **Decision**: Supabase Row-Level Security (RLS) is the authoritative authorization gate. Server-side TypeScript checks (`requireAdmin()`, `requireUser()`) layer on top for defense-in-depth and structured error responses — they never substitute for RLS.
- **Context**: Service-role key bypasses RLS; browser must never receive it. Debtors must not see other debtors' data even if a server bug exposes an incorrect `user_id`.
- **Consequences**:
  - Pro: Even if a server action has a logic bug, the DB refuses unauthorized reads/writes at the RLS layer.
  - Pro: Anon access returns empty sets, not errors, reducing information leakage.
  - Con: RLS policies must be maintained in sync with application logic; divergence creates subtle bugs. Mitigated by session 02's migration approach (RLS in its own `0002_rls.sql` migration with explicit policy names).
  - Con: Service-role client must be restricted to server context via a build-time guard (`typeof window !== 'undefined'` throws).

#### `docs/adr/0004-idempotent-jobs.md`
- **Status**: Accepted
- **Decision**: All scheduled jobs (installment generation, monthly accrual) are idempotent on their natural key: `(debt_id, sequence_number)` for installments; `(interest_debt_id, period, mode)` for accruals. Jobs use `INSERT ... ON CONFLICT DO NOTHING`.
- **Context**: Vercel cron jobs can be invoked multiple times for the same period due to retries, clock drift, or operator re-runs. A non-idempotent job would create duplicate installments or double-count accrued interest.
- **Consequences**:
  - Pro: Re-running any cron job for any historical period is safe; idempotency removes an entire class of production incidents.
  - Pro: Easy to backfill missed periods manually without guarding against double-execution.
  - Con: The unique constraint must be in place before the first job run; migration order matters (constraint established in `0001_init.sql`, `0003_unique_installment_seq.sql` adds it explicitly as a belt-and-suspenders guard).

## Design Decisions and Rationale

### Polymorphic `payment_applications.target_id`
The `target_type / target_id` pattern was chosen over two nullable FK columns (`installment_id`, `interest_debt_id`) because:
1. The allocation loop treats both targets uniformly; a shared `target_id` reduces branching.
2. Adding a third target type (unlikely, but possible) requires no schema change.
Risk: no DB-level FK constraint. Mitigation: application layer validates `target_type` before insert; unit tests cover both target types.

### `interest_rate` as `text` snapshot on `interest_debts`
Storing the rate string at conversion time rather than always reading from `settings` ensures that admin rate changes never retroactively alter existing obligations. This is consistent with standard loan agreement practice.

### `due_day` capped at 28
Months with fewer than 31 days (Feb, Apr, Jun, Sep, Nov) cause ambiguity if `due_day > 28`. Capping at 28 eliminates the February edge case entirely. Real-world loans issued on day 29/30/31 are rounded down; this is disclosed in the UI help text.

### `start_month` as `text ('YYYY-MM')`
Using a `text` column instead of `date` for month-precision fields avoids timezone ambiguity — a `date` stored as the 1st of the month is susceptible to off-by-one errors when the app server and DB are in different timezones. The application always constructs `due_date` via `start_month` + offset arithmetic in UTC.

### No `payment_applications` updates or deletes
`payment_applications` is an immutable append-only audit log. Any correction (e.g., a payment reversal) is handled by inserting a new `payments` row with a negative amount and corresponding `payment_applications` rows — this is a Phase 2+ feature, not yet in scope.
