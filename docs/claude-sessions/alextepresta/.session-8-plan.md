## Key Design Decisions

### 1. Migration Numbering

Session instructions request `0006_simulation_settings.sql` but `0006_phase2_apply_payment_fn.sql`
already exists from Session 7. Use `0007/0008/0009` instead. 0008 and 0009 are no-op migrations
with comments explaining the columns are pre-emptively in 0001.

Rationale: forward-only migration discipline — never edit published migrations. No-op files
preserve the numbered sequence without re-running DDL that would fail (columns already exist).

### 2. Simulated Mirror Creation: TypeScript Post-RPC, Not PL/pgSQL

The session instruction says "Extend `applyPayment.ts`", so keep the PL/pgSQL function stable
(no new migration updating `apply_payment`). Instead:

1. Call the existing `apply_payment` RPC.
2. Collect all `target_type='installment'` IDs from the returned applications.
3. Query `interest_debts WHERE source_installment_id IN (...) AND is_simulated=false AND mirror_of IS NULL`
   to find real debts just created by the RPC.
4. For each real debt, look up simulated rate (user override → global setting → real debt fallback).
5. Insert simulated twin: `is_simulated=true`, same `principal_minor`, `source_installment_id`,
   `debt_id`; `interest_rate` from step 4; `mirror_of = realDebt.id`.

Why no race condition risk: an installment transitions to `'converted'` atomically inside the PL/pgSQL
transaction. A second concurrent payment cannot target the same installment (it's no longer 'pending').
Therefore `source_installment_id`+`is_simulated=false` uniquely identifies a just-created debt.

Add a safe-guard query before inserting: check if a simulated row with `mirror_of = realDebt.id`
already exists (handles any unforeseen double-call scenario).

### 3. `runMonthlyAccrual` Mode Parameter

Add optional `mode: 'real' | 'simulated' = 'real'`. When `'simulated'`, query
`is_simulated=true` interest_debts. Accrual rows are tagged with that mode. The cron route
calls both sequentially (real first, then simulated). No changes to existing 'real' behavior.

### 4. `getDebtorOverview` — Rename + Split

Rename `interest_debt_balance_by_currency` → `real_balance_by_currency`.
Add `simulated_balance_by_currency` with a second DB query for `is_simulated=true` rows.
`total_owed_by_currency` keeps including real interest debt balances. Simulated balances
are never added to `total_owed`. All existing tests must be updated to use the new name.

### 5. UI Mode Toggle — URL Search Params Only

Mode is `?mode=simulada` in URL. Server components read `searchParams.mode`. No DB writes.
Toggle rendered as a Next.js `<Link>` (never `<form>` / action). Amber banner shown when
`mode === 'simulada'`. The `(mode?: string)` prop is read from `searchParams` passed to
the page component.

### 6. Scenarios Pages — Client-Side Computation

`ScenarioProjector` is a `'use client'` component. It imports `accrueOne` directly (no
`server-only` constraint on that module). Server component fetches initial interest_debt
data and passes it as props. Zero DB writes in the scenarios flow.

### 7. `simulated_annual_rate` Reading in TypeScript

The `settings.value` column is JSONB. The seed `'"0.24"'` returns as JS string `"0.24"` from
Supabase (JSONB already parsed). No quote-stripping needed for string values. The
`user_simulation_overrides.simulated_annual_rate` column is plain `text` for simplicity.

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| Migrations 0008/0009 are truly no-ops; Supabase migration runner might stumble on empty files | Write them as single-line SQL comments (valid SQL, non-empty). `-- No-op: column already in 0001_init.sql.` |
| `applyPayment` creates mirrors but the payment didn't trigger any conversion | Guard: only run mirror creation when `installmentTargetIds.length > 0` AND the real debt query returns rows |
| `user_simulation_overrides` may have 0 rows for most users | Rate lookup uses `?? globalRate ?? realDebt.interest_rate` chain — safe fallback |
| Renaming `interest_debt_balance_by_currency` breaks callers | Do a full codebase grep before coding; only the test file and the overview function itself reference it |
| Scenarios page imports `accrueOne` (uses `decimal.js`) client-side — bundle size | Acceptable; decimal.js is ~10 KB gzipped, already a prod dependency |
| `runMonthlyAccrual` default-parameter change breaks existing tests | Default stays `'real'`; all existing call sites (tests + cron route) continue to work without modification to their call syntax |
| DebtTimeline simulated events mixed with real events | Add `is_simulated` flag to `InterestDebtRow` and `AccrualRow` types; filter in `getDebtTimeline` based on mode |

### Step 4 — `applyPayment.ts`

Add helper `resolveSimulatedRate`:
```typescript
async function resolveSimulatedRate(
  adminClient: ReturnType<typeof createAdminClient>,
  debtorId: string,
  fallbackRate: string,
): Promise<string>
```
- Queries `user_simulation_overrides` for `user_id = debtorId`
- If found, returns `userOverride.simulated_annual_rate`
- Else queries `settings` for `key = 'simulated_annual_rate'`
- If found, returns `String(settingValue)` (already a string from JSONB)
- Else returns `fallbackRate` (real debt's rate)

Add logic after successful RPC call in `applyPayment`:
```typescript
const installmentIds = data.applications
  .filter((a) => a.target_type === 'installment')
  .map((a) => a.target_id)

if (installmentIds.length > 0) {
  const { data: newRealDebts } = await adminClient
    .from('interest_debts')
    .select('id, debt_id, source_installment_id, principal_minor, interest_rate')
    .in('source_installment_id', installmentIds)
    .eq('is_simulated', false)
    .is('mirror_of', null)   // real rows never have mirror_of set

  if (newRealDebts && newRealDebts.length > 0) {
    // Get debtor_id from the payment record
    const { data: paymentRow } = await adminClient
      .from('payments')
      .select('debtor_id')
      .eq('id', paymentId)
      .single()

    const debtorId = (paymentRow as { debtor_id: string } | null)?.debtor_id  // boundary: payment record shape

    for (const realDebt of newRealDebts) {
      // Guard: skip if mirror already exists (safety net for retries)
      const { data: existingMirror } = await adminClient
        .from('interest_debts')
        .select('id')
        .eq('mirror_of', realDebt.id)
        .maybeSingle()
      if (existingMirror) continue

      const simulatedRate = debtorId
        ? await resolveSimulatedRate(adminClient, debtorId, realDebt.interest_rate)
        : realDebt.interest_rate

      const { error: mirrorError } = await adminClient.from('interest_debts').insert({
        debt_id: realDebt.debt_id,
        source_installment_id: realDebt.source_installment_id,
        principal_minor: realDebt.principal_minor,
        current_balance_minor: realDebt.principal_minor,
        interest_rate: simulatedRate,
        is_simulated: true,
        mirror_of: realDebt.id,
      })

      if (mirrorError) {
        // Non-fatal: log but do not throw — real payment already applied successfully
        // boundary: error shape from Supabase PostgREST response
        console.error(`Failed to create simulated mirror for ${realDebt.id}: ${mirrorError.message}`)
      }
    }
  }
}
```

Note: `mirror_of` is on the SIMULATED row pointing at the REAL row. Real rows always have
`mirror_of = NULL`. This is why the check uses `.is('mirror_of', null)` on the real row to
find unmirrorable candidates (this verifies the real row was newly created, not that it has a mirror).
The actual duplicate guard is the `existingMirror` check (find simulated rows with `mirror_of = realDebt.id`).
