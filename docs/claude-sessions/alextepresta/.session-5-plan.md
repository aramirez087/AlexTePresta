# Session 5 Implementation Plan — Payment Pipeline (Phase 1 FIFO)

## Context Summary

Session 4 left us with a working debt + installment creation flow. The database schema (`0001_init.sql`) already has `payments` and `payment_applications` tables. RLS in `0002_rls.sql` already allows debtors to INSERT payments with `status='pending'`. The task for Session 5 is to build the full payment pipeline: submission (debtor), application (FIFO engine), approval (admin), and direct registration (admin), plus UI for each flow.

---

## Key Design Decisions

### 1. No PL/pgSQL for `applyPayment` — TypeScript transaction via admin client
The `accept_invite` and `create_debt_with_installments` functions are PL/pgSQL because Supabase JS has no native multi-statement transaction API for user-facing tables. **However**, the Supabase `service_role` client can run raw SQL via `adminClient.rpc('sql', ...)` or via PostgREST stored procedures.

**Decision**: Implement `applyPayment` as a PL/pgSQL function (`apply_payment`) similar to `accept_invite`. This is the only way to guarantee the `BEGIN … FOR UPDATE … COMMIT` transaction boundary with pessimistic locking across multiple table mutations within Supabase's JS client.

**Rationale**: A TypeScript loop calling separate Supabase queries outside a transaction cannot guarantee atomicity or proper `FOR UPDATE` locking (the lock is released between calls). A PL/pgSQL function runs in one transaction, matching the architecture doc spec exactly.

### 2. `submitPayment` as a next-safe-action server action (debtor-facing)
Follows the same `action.schema(...).action(...)` pattern as `createDebt`. Uses `requireUser()` (not `requireAdmin()`) to get the caller's ID. Validates the debtor has at least one open (non-cancelled) debt in the requested currency before inserting.

### 3. `approvePayment` and `registerPaymentDirect` as next-safe-action server actions (admin-only)
Both call `requireAdmin()`. `approvePayment` calls the `apply_payment` RPC. `registerPaymentDirect` inserts a payment with `created_by = admin.id` then immediately calls `apply_payment`.

### 4. Error types as plain `Error` subclasses (consistent with existing pattern)
`createDebt.ts` throws `new Error(...)` which `handleServerError` in `safe-action.ts` forwards as `serverError`. Same approach here. The PL/pgSQL function uses `RAISE EXCEPTION 'PaymentExcessError: ...'` so the TS layer can detect by checking `error.message.startsWith('PaymentExcessError')`.

### 5. Phase 1 only — no partial-payment conversion
`applyPayment` Phase 1: if `remaining_minor > 0` after the FIFO loop, rollback and raise `PaymentExcessError`. Phase 2 (interest debt creation) is deferred to Session 07.

### 6. `handleServerError` already forwards `e.message` — no change to `safe-action.ts` needed

## Order of Operations

### Step 1 — Migration: `0005_apply_payment_fn.sql`

Create `apply_payment(p_payment_id uuid) RETURNS jsonb` as a SECURITY DEFINER function callable only by `service_role`.

**Algorithm** (Phase 1 FIFO, based on `docs/architecture/payment-pipeline.md`):

```sql
CREATE OR REPLACE FUNCTION public.apply_payment(p_payment_id uuid)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_payment         payments%ROWTYPE;
  v_remaining       bigint;
  v_installment     installments%ROWTYPE;
  v_to_apply        bigint;
  v_applications    jsonb := '[]'::jsonb;
  v_app_row         jsonb;
BEGIN
  -- Lock the payment row; assert status = 'pending'
  SELECT * INTO v_payment FROM payments WHERE id = p_payment_id FOR UPDATE;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'PaymentNotFoundError: payment % not found', p_payment_id;
  END IF;
  IF v_payment.status <> 'pending' THEN
    RAISE EXCEPTION 'PaymentAlreadyAppliedError: payment % is not pending', p_payment_id;
  END IF;

  v_remaining := v_payment.amount_minor;

  -- FIFO: pending installments for this debtor's debts in matching currency
  FOR v_installment IN
    SELECT i.*
    FROM installments i
    JOIN debts d ON d.id = i.debt_id
    WHERE d.debtor_id = v_payment.debtor_id
      AND d.currency  = v_payment.currency
      AND i.status    = 'pending'
    ORDER BY i.due_date ASC, i.sequence_number ASC
    FOR UPDATE
  LOOP
    EXIT WHEN v_remaining = 0;

    v_to_apply := LEAST(v_remaining, v_installment.remaining_amount_minor);

    INSERT INTO payment_applications
      (payment_id, target_type, target_id, applied_amount_minor)
    VALUES
      (p_payment_id, 'installment', v_installment.id, v_to_apply)
    RETURNING jsonb_build_object(
      'target_id', target_id,
      'target_type', target_type,
      'applied_amount_minor', applied_amount_minor
    ) INTO v_app_row;

    v_applications := v_applications || jsonb_build_array(v_app_row);

    UPDATE installments
    SET remaining_amount_minor = remaining_amount_minor - v_to_apply,
        status = CASE WHEN remaining_amount_minor - v_to_apply = 0 THEN 'paid' ELSE status END
    WHERE id = v_installment.id;

    v_remaining := v_remaining - v_to_apply;
  END LOOP;

  -- Phase 1: reject excess
  IF v_remaining > 0 THEN
    RAISE EXCEPTION 'PaymentExcessError: payment % has % minor units unallocated',
      p_payment_id, v_remaining;
  END IF;

  UPDATE payments
  SET status = 'approved', applied_at = now()
  WHERE id = p_payment_id;

  RETURN jsonb_build_object(
    'applications', v_applications,
    'leftover_minor', 0
  );
END;
$$;

REVOKE ALL ON FUNCTION public.apply_payment(uuid) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.apply_payment(uuid) TO service_role;
```

**Important SQL note**: The `RETURNING` clause on the `INSERT INTO payment_applications` selects from the inserted row. But to build a JSON object with the column values, we need to use the correct approach. An alternative is to insert, then build the JSON from local variables. Let me revise to avoid the `RETURNING … INTO` ambiguity:

```sql
-- Insert application row
INSERT INTO payment_applications
  (payment_id, target_type, target_id, applied_amount_minor)
VALUES
  (p_payment_id, 'installment', v_installment.id, v_to_apply);

v_app_row := jsonb_build_object(
  'target_id', v_installment.id,
  'target_type', 'installment',
  'applied_amount_minor', v_to_apply
);
v_applications := v_applications || jsonb_build_array(v_app_row);
```

Also note: the `UPDATE installments SET status = CASE WHEN remaining_amount_minor - v_to_apply = 0 THEN 'paid' ELSE status END WHERE id = ...` uses the old value of `remaining_amount_minor` in the CASE condition. Since the UPDATE references the column before modification, `remaining_amount_minor - v_to_apply` correctly computes the new value. This is correct PostgreSQL behavior.

### Step 3 — `src/lib/domain/payments/applyPayment.ts`

Internal utility (not a server action). Called by `approvePayment` and `registerPaymentDirect`.

```typescript
import 'server-only'
import type { createAdminClient } from '@/lib/supabase/admin'

export type ApplicationResult = {
  target_id: string
  target_type: 'installment' | 'interest_debt'
  applied_amount_minor: number
}

export type ApplyPaymentResult = {
  applications: ApplicationResult[]
  leftover_minor: 0
}

export async function applyPayment(
  adminClient: ReturnType<typeof createAdminClient>,
  paymentId: string,
): Promise<ApplyPaymentResult> {
  const { data, error } = await adminClient.rpc('apply_payment', {
    p_payment_id: paymentId,
  })

  if (error) {
    if (error.message.includes('PaymentExcessError')) {
      throw new Error('El monto del pago excede las obligaciones pendientes')
    }
    if (error.message.includes('PaymentAlreadyAppliedError')) {
      throw new Error('Este pago ya fue procesado')
    }
    if (error.message.includes('PaymentNotFoundError')) {
      throw new Error('Pago no encontrado')
    }
    throw new Error(`Error al aplicar pago: ${error.message}`)
  }

  // boundary: RPC returns Json (unknown); function contract guarantees ApplyPaymentResult shape
  return data as ApplyPaymentResult
}
```

### Step 4 — `src/lib/domain/payments/submitPayment.ts`

Debtor server action. Uses `requireUser()`. Validates debtor has open debt in that currency.

```typescript
'use server'
import { z } from 'zod'
import { action } from '@/lib/safe-action'
import { createAdminClient } from '@/lib/supabase/admin'
import { requireUser } from '@/lib/auth/session'

const submitPaymentSchema = z.object({
  currency: z.enum(['CRC', 'USD']),
  amount_minor: z.number().int().positive().max(Number.MAX_SAFE_INTEGER),
  notes: z.string().max(500).optional(),
})

export const submitPayment = action
  .schema(submitPaymentSchema)
  .action(async ({ parsedInput: { currency, amount_minor, notes } }) => {
    const user = await requireUser()
    const admin = createAdminClient()

    // Validate debtor has at least one active debt in this currency
    const { data: debts } = await admin
      .from('debts')
      .select('id')
      .eq('debtor_id', user.id)
      .eq('currency', currency)
      .eq('status', 'active')
      .limit(1)

    if (!debts || debts.length === 0) {
      throw new Error(`No tienes deudas activas en ${currency}`)
    }

    const { data, error } = await admin
      .from('payments')
      .insert({
        debtor_id: user.id,
        currency,
        amount_minor,
        status: 'pending',
        created_by: user.id,
        notes: notes ?? null,
      })
      .select('id')
      .single()

    if (error) throw new Error(`Error al registrar pago: ${error.message}`)

    // boundary: PostgREST select returns unknown shape; function contract guarantees id field
    const row = data as { id: string }
    return { paymentId: row.id }
  })
```

## Risks and Mitigations

### Risk 1: PL/pgSQL UPDATE with computed CASE on same column
The `UPDATE installments SET remaining_amount_minor = remaining_amount_minor - v_to_apply, status = CASE WHEN remaining_amount_minor - v_to_apply = 0 THEN 'paid' ELSE status END` — in PostgreSQL, the CASE in SET references the column's value *before* the update (the old row). So `remaining_amount_minor - v_to_apply` in the CASE correctly computes the new value.

**Mitigation**: Verify with a local test. Alternative: use a local variable:
```sql
DECLARE v_new_remaining bigint;
...
v_new_remaining := v_installment.remaining_amount_minor - v_to_apply;
UPDATE installments
  SET remaining_amount_minor = v_new_remaining,
      status = CASE WHEN v_new_remaining = 0 THEN 'paid' ELSE status END
WHERE id = v_installment.id;
```
Use the variable approach — it's unambiguous.

### Risk 2: Loop variable `v_installment.remaining_amount_minor` is stale
The loop variable is set at `SELECT … FOR UPDATE` time. If the installment has already been partially paid in the same transaction (impossible since we're in a single loop iteration per installment), this could be wrong. Since each installment is processed once per loop iteration, `v_installment.remaining_amount_minor` is always the value at the start of the transaction (no concurrent modification possible due to FOR UPDATE). This is correct.

### Risk 3: RPC returns `Json` (unknown) type
`adminClient.rpc('apply_payment', ...)` returns `{ data: Json; error: ... }`. The `as ApplyPaymentResult` cast is a trust boundary. Ensure the cast comment is present.

### Risk 4: `(debtor)` route group and middleware
The `/pay` route must be protected. The middleware update (`pathname.startsWith('/pay')`) must be done carefully to not accidentally protect non-debtor paths. Verify the matcher pattern.

### Risk 5: `amount_minor` Zod validation uses `z.number()` not `z.bigint()`
Same pattern as `createDebt`. Amounts are passed as `number` through the form (after ×100), capped at `MAX_SAFE_INTEGER`. The DB column is `bigint` but PostgREST accepts a JS `number` for it. Keep consistent with existing pattern.

### Risk 6: Admin payments page — no `/admin/debtors/[id]` page yet
The register-payment page references a non-existent parent page. Create the register-payment page with a back link to `/admin/payments` or `/admin/debts/new` for now.

## Notes on Supabase Transaction Limitation

Supabase JS does not expose a `BEGIN` / `COMMIT` API. All multi-statement atomic work must go through PL/pgSQL functions with `LANGUAGE plpgsql`. The `apply_payment` function follows the same pattern as `accept_invite` (Session 3) and `create_debt_with_installments` (Session 4). This is the established project pattern.

The function is granted only to `service_role`, called via `adminClient.rpc(...)`, and is `SECURITY DEFINER` so it executes with superuser-equivalent rights on the tables. This matches the existing migration security model.

## Exit Criteria Verification

1. **Debtor pays exactly one installment** → `apply_payment` sets `installment.status='paid'`, `installment.remaining_amount_minor=0`, inserts one `payment_applications` row with `applied_amount_minor = installment.amount_minor`. `payment.status='approved'`.

2. **Debtor pays 2× one installment (overflow)** → First installment paid, second installment `remaining_amount_minor` reduced. Two `payment_applications` rows. Full audit: sum of `applied_amount_minor` across `payment_applications` for this `payment_id` equals `payment.amount_minor`.

3. **Admin direct registration = debtor submit + admin approve** → Both flows call the same `apply_payment` RPC. The only difference is `created_by`: debtor's own ID vs admin's ID. All other columns identical. DB state identical after both flows.
