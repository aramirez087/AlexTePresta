## Step 1: `src/lib/format/money.ts`

### Design

```typescript
const MINOR_DIVISORS: Record<'CRC' | 'USD', number> = { CRC: 100, USD: 100 }
const LOCALES: Record<'CRC' | 'USD', string> = { CRC: 'es-CR', USD: 'en-US' }

export function formatMoney(amountMinor: bigint, currency: 'CRC' | 'USD'): string {
  const divisor = MINOR_DIVISORS[currency]
  // safe: domain amounts never exceed Number.MAX_SAFE_INTEGER
  // (₡90 quadrillion would be required to overflow — well beyond any personal loan)
  const major = Number(amountMinor) / divisor
  return new Intl.NumberFormat(LOCALES[currency], {
    style: 'currency',
    currency,
    minimumFractionDigits: 2,
  }).format(major)
}
```

**Precision note:** `Number(amountMinor)` is safe because loan amounts in this domain cap at hundreds of millions of CRC (max ~₡500M = 50,000,000,000 minor units), well below `Number.MAX_SAFE_INTEGER` (9,007,199,254,740,991). A comment at the boundary explicitly names this invariant.

## Step 2: `src/lib/format/date.ts`

### Design

```typescript
const CR_TZ = 'America/Costa_Rica'
const CR_DATE_FMT = new Intl.DateTimeFormat('es-CR', {
  year: 'numeric',
  month: 'long',
  day: 'numeric',
  timeZone: CR_TZ,
})

export function formatDate(d: Date): string {
  return CR_DATE_FMT.format(d)
}

export function daysUntil(d: Date): number {
  // Compute calendar-day difference in Costa Rica timezone to avoid DST errors
  const isoFormatter = new Intl.DateTimeFormat('en-CA', { timeZone: CR_TZ })
  const todayStr = isoFormatter.format(new Date())     // YYYY-MM-DD
  const targetStr = isoFormatter.format(d)             // YYYY-MM-DD
  const todayMs = new Date(todayStr + 'T00:00:00').getTime()
  const targetMs = new Date(targetStr + 'T00:00:00').getTime()
  return Math.round((targetMs - todayMs) / 86_400_000)
}
```

**Timezone invariant:** `America/Costa_Rica` is UTC-6 with no daylight saving time. This means the DST edge case tests verify that `en-CA` formatter correctly localizes dates even when the JS Date object is in UTC or another timezone.

## Step 3: `src/lib/domain/views/getDebtTimeline.ts`

### Design

This is a **pure function** — it takes already-fetched data rows and returns sorted events. No Supabase calls inside. The caller (page) fetches the data and passes it in, making the function fully testable without mocks.

```typescript
export type TimelineEventKind =
  | 'installment_due'
  | 'payment_received'
  | 'payment_application'

export type TimelineEvent = {
  kind: TimelineEventKind
  date: Date
  amount_minor: bigint
  currency: 'CRC' | 'USD'
  status: string
  ref_id: string
  meta?: Record<string, unknown>
}

type InstallmentRow = {
  id: string
  due_date: string
  amount_minor: number
  remaining_amount_minor: number
  status: string
  sequence_number: number
}

type PaymentRow = {
  id: string
  created_at: string
  applied_at: string | null
  amount_minor: number
  currency: string
  status: string
  notes: string | null
}

type ApplicationRow = {
  id: string
  payment_id: string
  target_id: string
  applied_amount_minor: number
  created_at: string
}

export type GetDebtTimelineParams = {
  installments: InstallmentRow[]
  payments: PaymentRow[]
  applications: ApplicationRow[]
  currency: 'CRC' | 'USD'
}

export function getDebtTimeline(params: GetDebtTimelineParams): TimelineEvent[] {
  const events: TimelineEvent[] = []

  for (const inst of params.installments) {
    events.push({
      kind: 'installment_due',
      // Use noon UTC to keep the date stable across timezones for comparison
      date: new Date(inst.due_date + 'T12:00:00Z'),
      amount_minor: BigInt(inst.amount_minor),
      currency: params.currency,
      status: inst.status,
      ref_id: inst.id,
      meta: {
        sequence_number: inst.sequence_number,
        remaining_amount_minor: BigInt(inst.remaining_amount_minor),
      },
    })
  }

  for (const pmt of params.payments) {
    events.push({
      kind: 'payment_received',
      date: new Date(pmt.created_at),
      amount_minor: BigInt(pmt.amount_minor),
      currency: pmt.currency as 'CRC' | 'USD',
      status: pmt.status,
      ref_id: pmt.id,
      meta: { notes: pmt.notes, applied_at: pmt.applied_at },
    })
  }

  for (const app of params.applications) {
    events.push({
      kind: 'payment_application',
      date: new Date(app.created_at),
      amount_minor: BigInt(app.applied_amount_minor),
      currency: params.currency,
      status: 'applied',
      ref_id: app.id,
      meta: { payment_id: app.payment_id, target_id: app.target_id },
    })
  }

  return events.sort((a, b) => a.date.getTime() - b.date.getTime())
}
```

**No `'use server'` directive** — this is a pure domain function, not a server action.

## Step 4: `src/lib/domain/views/getDebtorOverview.ts`

### Design

This function queries Supabase using the admin client. It returns a structured overview for a single debtor.

```typescript
import 'server-only'
import type { createAdminClient } from '@/lib/supabase/admin'

export type DebtSummary = {
  id: string
  description: string | null
  currency: 'CRC' | 'USD'
  total_amount_minor: bigint
  total_installments: number
  installment_amount_minor: bigint
  status: string
  next_pending_installment: {
    id: string
    due_date: string
    remaining_amount_minor: bigint
    sequence_number: number
  } | null
}

export type DebtorOverview = {
  debts: DebtSummary[]
  total_owed_by_currency: { CRC: bigint; USD: bigint }
  total_paid_by_currency: { CRC: bigint; USD: bigint }
  next_installment: {
    due_date: string
    amount_minor: bigint
    currency: 'CRC' | 'USD'
  } | null
  status: 'al_dia' | 'atrasado'
}

export async function getDebtorOverview(
  adminClient: ReturnType<typeof createAdminClient>,
  debtorId: string,
): Promise<DebtorOverview>
```

**Implementation logic:**
1. Fetch all `debts` where `debtor_id = debtorId AND status = 'active'`
2. For each debt, fetch `installments` where `debt_id = debt.id AND status != 'paid'`, ordered by `due_date ASC`, limit 1 for the next pending installment
3. Compute `total_owed_by_currency`: sum of `installment.remaining_amount_minor` per currency across all active pending installments
4. Compute `total_paid_by_currency`: sum of `(amount_minor - remaining_amount_minor)` for all installments (including paid ones)
5. `status = 'atrasado'` if any pending installment has `due_date < today` (compare as ISO date strings)
6. `next_installment` = earliest pending installment across all debts

**Query strategy:** Fetch all pending installments in a single query (not per-debt) to avoid N+1:
```
adminClient
  .from('installments')
  .select('*, debt:debts!inner(id, currency, debtor_id, status)')
  .eq('debt.debtor_id', debtorId)
  .eq('debt.status', 'active')
  .neq('status', 'paid')
  .order('due_date', { ascending: true })
```

Then group by debt in TypeScript.

### Design Decision: Server Component

`DebtTimeline` is a **server component** (no `'use client'`). It accepts `TimelineEvent[]` (which contains `bigint`) directly from server pages. Since there is no server→client serialization boundary, `bigint` props are legal. The component has no event handlers and no client-side interactivity, making RSC the correct choice.

### Data fetching

The page fetches all debtors and per-debtor stats in one pass:

```typescript
export const dynamic = 'force-dynamic'

export default async function AdminHomePage() {
  await requireAdmin()
  const admin = createAdminClient()

  // All users with role='debtor'
  const { data: debtors } = await admin
    .from('users')
    .select('id, email')
    .eq('role', 'debtor')
    .order('email')

  // For each debtor: compute overview
  const overviews = await Promise.all(
    (debtors ?? []).map(async (d) => {
      const overview = await getDebtorOverview(admin, d.id)
      // Count overdue installments (already derivable from overview data)
      // Count pending payments
      const { count: pendingPayments } = await admin
        .from('payments')
        .select('id', { count: 'exact', head: true })
        .eq('debtor_id', d.id)
        .eq('status', 'pending')
      
      return { debtor: d, overview, pendingPayments: pendingPayments ?? 0 }
    })
  )
}
```

**Alert counts:**
- Overdue installments: count from `getDebtorOverview` by iterating `debts` and checking `next_pending_installment.due_date < today`
- Pending payment approvals: separate count query

**Alternative for overdue count:** Instead of deriving from overview, add a helper to count overdue installments directly. Since `getDebtorOverview` already fetches pending installments, extract overdue count from the overview result:
```typescript
const overdueCount = overview.debts.reduce((sum, debt) => {
  const next = debt.next_pending_installment
  if (next && next.due_date < todayStr) return sum + 1
  return sum
}, 0)
```
This avoids an extra query.

    .select('id, email, role')
    .eq('id', id)
    .single()

  if (!debtor || debtor.role !== 'debtor') notFound()

  const overview = await getDebtorOverview(admin, id)

  // Fetch timeline data: all installments, payments, applications
  // for this debtor across all active debts
  const { data: allInstallments } = await admin
    .from('installments')
    .select('*, debt:debts!inner(debtor_id, currency, status)')
    .eq('debt.debtor_id', id)
    .eq('debt.status', 'active')
    .order('due_date')

  const { data: allPayments } = await admin
    .from('payments')
    .select('*')
    .eq('debtor_id', id)
    .in('status', ['pending', 'approved'])
    .order('created_at')

  const paymentIds = (allPayments ?? []).map(p => p.id)
  const { data: allApplications } = paymentIds.length > 0
    ? await admin
        .from('payment_applications')
        .select('*')
        .in('payment_id', paymentIds)
        .order('created_at')
    : { data: [] }

  // For timeline, determine currency (use first debt's currency, or 'CRC' as fallback)
  // In multi-currency scenario, render separate timelines per currency
}

## Key Design Decisions

### 1. `getDebtTimeline` is a pure function (no Supabase calls)

**Rationale:** Pure functions are trivially testable without mocks. The pages that use the timeline are already making Supabase calls; they pass the fetched data to `getDebtTimeline`. This separation makes the domain logic independently verifiable.

**Trade-off:** Pages have slightly more data-fetching boilerplate. Acceptable for this domain size.

### 2. `DebtTimeline` is a server component

**Rationale:** The component accepts `bigint` from domain types. Server components have no serialization boundary, so `bigint` flows through cleanly. The component is display-only (no event handlers), making RSC the correct model.

**Trade-off:** The component cannot have client-side interactions (e.g., expand/collapse). For Phase 1 this is fine. Phase 2 could wrap it in a client component if needed.

### 3. Debtor home stays at `/app`, not `/`

**Rationale:** `src/app/(debtor)/page.tsx` would conflict with `src/app/page.tsx` at URL `/`. The existing middleware protects `/app` and redirects unauthenticated admin users back to `/app`. Moving the debtor home to `/` would require updating middleware and breaking the admin redirect flow.

**Trade-off:** Slight inconsistency with the session plan's stated file path. Documented in handoff.

### 4. Admin home replaces the invites redirect

**Rationale:** The admin home should show operational information (debtor list, alerts) rather than dumping users in the invites page. Navigation links to invites, payments, and new debt creation are added to the admin home layout.

**Trade-off:** If there's a bookmark to `/admin`, it now shows the debtor list instead of invites. Better UX.

### 5. `export const dynamic = 'force-dynamic'` on all new Supabase pages

**Rationale:** Fixes the Session 5 open issue where the build fails during static prerendering. All pages that call Supabase must opt out of static generation.

## Risks and Mitigations

### Risk 1: `getDebtorOverview` N+1 queries in admin home

**Problem:** Admin home calls `getDebtorOverview` for each debtor. With 10 debtors, that's 10+ Supabase round trips per page load.

**Mitigation:** For Phase 1 (personal loan app with few debtors, max ~5-10), this is acceptable. Document it in the handoff. Phase 2 can replace with a single aggregate query or Supabase view.

**Alternative considered:** Single query joining all tables — complex and harder to maintain. Deferred.

### Risk 2: Multi-currency timeline rendering

**Problem:** A debtor with CRC and USD debts has payments in different currencies. The `getDebtTimeline` function accepts a single `currency`. Mixing currencies in one timeline would show incorrect formatting.

**Mitigation:** Group by currency in the page component. Render separate `<DebtTimeline>` per currency. If only one currency exists, render one section without a currency header.

### Risk 3: `daysUntil` test flakiness (depends on "today")

**Problem:** `daysUntil` calls `new Date()` internally, making tests time-sensitive.

**Mitigation:** Use `vi.useFakeTimers()` and `vi.setSystemTime()` in the test file to pin "today" to a known date. Reset after each test with `vi.useRealTimers()`.

### Risk 4: Snapshot test brittleness (formatMoney output locale)

**Problem:** Snapshot tests for `DebtTimeline` will include formatted money strings. These might differ across Node.js versions or CI environments (ICU data differences).

**Mitigation:** Use targeted assertions (`.toHaveTextContent('₡')`) for critical labels rather than full snapshots. Reserve snapshots for structural/accessibility tests. Include a note in the handoff about potential ICU divergence.

### Risk 5: `Intl.NumberFormat` ICU data in test environment

**Problem:** The vitest test environment (Node.js) may not have full ICU data, causing `es-CR` locale to produce different output than production.

**Mitigation:** Test for presence of key strings (currency symbol, approximate numeric value) rather than exact formatted output. Use `expect(result).toContain('₡')` and `expect(result).toContain('1,478')` rather than exact string equality.

### Risk 6: `payment_applications` may be empty when `paymentIds` is empty

**Problem:** `adminClient.from('payment_applications').select('*').in('payment_id', [])` — the Supabase JS client behavior with empty `.in()` arrays is undefined/may error.

**Mitigation:** Guard with `if (paymentIds.length === 0) return { data: [] }` before the query. This pattern is already anticipated in the Step 9 pseudocode above.

## `revalidatePath` for Real-time Alert Decrement

The exit criterion says "counts decrement after approval without manual refresh (server action revalidates)."

The `approvePayment` server action in `src/lib/domain/payments/approvePayment.ts` currently returns the result but doesn't call `revalidatePath`. To make the admin home update after approval:

**Add to `approvePayment.ts`** (or in the approve-button's `onSuccess` handler):
```typescript
import { revalidatePath } from 'next/cache'
// After successful apply:
revalidatePath('/admin')
revalidatePath('/admin/payments')
```

Since `approvePayment.ts` is a server action and `revalidatePath` is available in server actions, this is the correct hook point. The `approve-button` client component triggers the action, and Next.js automatically re-fetches the invalidated paths.

**Note:** This requires modifying `src/lib/domain/payments/approvePayment.ts` which is from Session 5. The change is minimal (2 lines) and additive. It does not affect the existing 80 tests since the mock already allows the action to succeed.
