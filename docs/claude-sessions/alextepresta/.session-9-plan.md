## Key Design Decisions

### D1: Coverage tool — `@vitest/coverage-v8`
V8 native coverage (no instrumentation), standard for Node.js, matches the installed vitest version `^3.2.4`. Add `"@vitest/coverage-v8": "^3.2.4"` to devDependencies.

### D2: CI env vars — hardcoded local demo values
For `NEXT_PUBLIC_*` keys and `SUPABASE_SERVICE_ROLE_KEY`, use the default local Supabase demo JWTs from `.env.local.example`. These are publicly documented by Supabase, work only against a local instance, and are safe to commit. Using GitHub secrets for these demo values adds friction with no security benefit.

The production service role key should be set as a GitHub secret (`${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}`), but for CI (build-only, no real Supabase connection), the demo value suffices.

### D3: CI does NOT run Supabase DB migrations
CI pipeline runs build, type-check, and unit tests only. Unit tests use mocked Supabase clients — no live database needed. Starting Supabase in CI adds complexity and latency with no proportional benefit. Migration testing is done locally via `supabase db reset`. Trade-off accepted: CI won't catch SQL errors; migrations are reviewed before merge.

### D4: Fix `is_simulated` propagation in Session 9
This is a listed "next step" from Session 08 and is small and safe. Without it, the amber dashed border in `DebtTimeline.tsx` never fires in production. Fix it here to complete the simulation feature end-to-end.

### D5: Regenerate `database.types.ts` after `supabase db reset`
Removing the `as unknown as` cast in `applyPayment.ts` requires the typed client to have the `Database` generic. After regeneration, check `src/lib/supabase/admin.ts` — if `createClient` is called with the `Database` generic, the cast can be removed cleanly. If not, keep the cast and document it.

**4c. Key risk: `bigint` serialization across server/client boundary**

`DebtTimeline.tsx` receives `events: TimelineEvent[]` which contains `amount_minor: bigint`. If `DebtTimeline` has `'use client'`, Next.js will throw a serialization error at build time because `bigint` cannot be JSON-serialized for RSC props.

**Investigation step:** Read `src/components/timeline/DebtTimeline.tsx` and check for `'use client'` directive.

- **If no `'use client'`** (server component): no fix needed, `bigint` is rendered server-side.
- **If `'use client'` is present**: fix required. Options:
  1. Remove `'use client'` if no browser APIs are used (preferred — keeps bigint domain types intact)
  2. Add a `SerializedTimelineEvent` type with `amount_minor: string`; convert at the page boundary before passing to the client component; update `DebtTimeline` props accordingly

**4d. If build fails for other reasons**
- Missing env vars → verify `.env.local` is complete
- Import errors → check `@/` alias resolves via `tsconfig.json`
- `server-only` boundary violation → a client component imports a server-only module; find and remove the `'use client'` directive or restructure

### Phase 8: Timeline `is_simulated` Fix

**8a. Update `src/lib/domain/views/getDebtTimeline.ts`**

Change `InterestDebtRow` type (add optional `is_simulated` field):

```typescript
type InterestDebtRow = {
  id: string
  source_installment_id: string | null
  principal_minor: number
  current_balance_minor: number
  interest_rate: string
  created_at: string
  is_simulated?: boolean  // propagated to meta.simulated for DebtTimeline amber rendering
```

Change the `interest_debt_created` event creation (in the `for (const idb of params.interest_debts ?? [])` loop):
```typescript
events.push({

  date: new Date(idb.created_at),
  amount_minor: BigInt(idb.principal_minor),
  currency: params.currency,
  status: 'active',
  ref_id: idb.id,

    source_installment_id: idb.source_installment_id,
    interest_rate: idb.interest_rate,
    current_balance_minor: BigInt(idb.current_balance_minor),
    simulated: idb.is_simulated ?? false,  // enables amber dashed border in DebtTimeline

```

**8b. Update `src/lib/domain/views/__tests__/getDebtTimeline.test.ts`**

Add 2 new tests after the existing `interest_debt_created` tests:

```typescript
it('sets meta.simulated=true for simulated interest_debt', () => {
  const params = makeParams({
    interest_debts: [
      {
        id: IDEBT_ID_1,
        source_installment_id: INST_ID_1,
        principal_minor: 47875,
        current_balance_minor: 47875,
        interest_rate: '0.36',
        created_at: '2024-06-05T10:02:00Z',
        is_simulated: true,
      },
    ],
  })
  const events = getDebtTimeline(params)
  expect(events[0].meta?.simulated).toBe(true)
})

it('sets meta.simulated=false for real interest_debt', () => {
  const params = makeParams({
    interest_debts: [
      {
        id: IDEBT_ID_1,
        source_installment_id: INST_ID_1,
        principal_minor: 47875,
        current_balance_minor: 47875,
        interest_rate: '0.24',
        created_at: '2024-06-05T10:02:00Z',
        is_simulated: false,
      },
    ],
  })
  const events = getDebtTimeline(params)
  expect(events[0].meta?.simulated).toBe(false)
})
```

Note: Existing tests that pass `interest_debts` without `is_simulated` continue to work because:
- The field is optional (`is_simulated?: boolean`)
- The event sets `simulated: idb.is_simulated ?? false` (defaults to false when absent)
- TypeScript structural typing allows passing objects with additional properties to the existing tests

## Risks and Mitigations

### Risk 1: `bigint` serialization across server/client boundary in `DebtTimeline`
- **Probability:** Medium — `DebtTimeline` may have `'use client'` if animation or interactive logic was added
- **Impact:** High — blocks `npm run build`
- **Detection:** Read `src/components/timeline/DebtTimeline.tsx` in Phase 4
- **Mitigation A (preferred):** Remove `'use client'` if no browser APIs are used
- **Mitigation B:** Create `SerializedTimelineEvent` with `amount_minor: string`; convert at page boundary; `bigint` stays server-side

### Risk 2: `applyPayment.ts` cast removal causes type errors
- **Probability:** Medium — depends on whether `admin.ts` uses `Database` generic
- **Impact:** Low — keep cast if needed, no functional change
- **Mitigation:** Check `admin.ts` before removing cast; if not typed, add comment and leave cast

### Risk 3: `npm run build` fails due to missing env vars
- **Probability:** Low — all pages use `force-dynamic`; no static generation
- **Impact:** Medium — blocking
- **Mitigation:** Verify `.env.local` is complete; add `export const dynamic = 'force-dynamic'` to any page missing it

### Risk 4: Migration 0007 fails on `supabase db reset`
- **Probability:** Very low — migration is simple `CREATE TABLE` with FK to `users`
- **Impact:** Medium — blocks type regeneration
- **Mitigation:** Verify FK ordering (0001 creates `users` table; 0007 FK references it — correct order)

### Risk 5: Coverage below target
- **Probability:** Low — domain code is well-tested per Session 08 (175 tests)
- **Impact:** Low — coverage is informational, not a hard gate
- **Mitigation:** Document actual coverage; no minimum threshold enforced in this session

### Risk 6: Smoke test fails due to local OAuth configuration
- **Probability:** Medium — Google OAuth requires valid credentials
- **Impact:** Low — smoke test can be partially completed; admin→invite→debt→payment flow can be verified even without OAuth if local email auth is used
- **Mitigation:** Use Supabase local email/password auth for the smoke test if Google OAuth not configured; document this in the report
